# CR10: Owner Stamping — Auto-populate Owner from Authenticated User

## Summary

When creating a new ClientContact or ClientCorporation via the MCP server, the `owner` field must currently be provided explicitly by the caller. CR10 changes this: if the caller does not provide an `owner`, the server automatically sets it to the authenticated user's Bullhorn CorporateUser (resolved via CR9). If the caller explicitly provides an `owner`, their value is honoured without override.

## Problem

Today, `create_contact` requires `owner` in the input fields and returns an error if it is missing. This creates friction in two scenarios:

1. **Consultants using chat agents** must remember to include their own CorporateUser ID or name in every create request, even though they are already authenticated and identifiable.
2. **Automated agents (e.g. twin.so)** must be configured with a specific owner value for every write operation, which duplicates identity information that the server already has.

## Required Changes

### 1. Make `owner` optional in `create_contact`

**File:** `src/bullhorn_mcp/server.py`

Currently, `create_contact` validates that `owner` is present in the `fields` dict and returns an error if missing. Change this:

- If `owner` is present in `fields`: proceed as today. The existing `resolve_owner()` flow handles string names and `{"id": int}` dicts.
- If `owner` is absent from `fields`: call `resolve_caller()` from CR9's `identity.py` to get the authenticated user's CorporateUser, then set `fields["owner"] = {"id": caller["id"]}`.

The auto-populated owner is set *before* the duplicate check and *before* the `client.create()` call, so it flows through the existing pipeline unchanged.

```python
# Pseudocode for the change in create_contact:

# BEFORE (current):
if "owner" not in fields:
    return format_response({"error": "owner is required"})

# AFTER (CR10):
if "owner" not in fields:
    caller = resolve_caller()
    fields["owner"] = {"id": caller["id"]}
```

### 2. Make `owner` optional in `create_company`

**File:** `src/bullhorn_mcp/server.py`

Apply the same pattern. If `create_company` currently validates for `owner`, make it optional with the same fallback to `resolve_caller()`. If it does not currently require `owner`, add the auto-population when `owner` is absent.

### 3. No change to `bulk_import`

`bulk_import` has its own owner resolution logic via `BulkImporter`. Owner stamping for bulk imports should be addressed separately if needed — CR10 only covers `create_contact` and `create_company`.

### 4. No change to `update_record`

Updating the `owner` field on an existing record is a deliberate action. CR10 does not auto-populate `owner` on updates — it only applies to record creation.

### 5. Caller-provided owner always wins

If the caller explicitly provides an `owner` in the `fields` dict (whether as a string name like `"Maryrose Lyons"` or as `{"id": 42}`), that value is used. The resolved identity is a sensible default, not a policy enforcement. Consultants may legitimately create records owned by colleagues.

The logic is straightforward: check whether `"owner"` is a key in `fields`. If yes, use the existing `resolve_owner()` path. If no, call `resolve_caller()` and inject it.

### 6. Error handling when identity resolution fails

If `resolve_caller()` raises `IdentityResolutionError` (no token, no email match, etc.), the tool should return a formatted error response to the caller, not crash. The error message from `IdentityResolutionError` is descriptive enough to surface directly.

```python
# Pseudocode:
if "owner" not in fields:
    try:
        caller = resolve_caller()
        fields["owner"] = {"id": caller["id"]}
    except IdentityResolutionError as e:
        return format_response({
            "error": "identity_resolution_failed",
            "message": str(e),
            "hint": "Provide an explicit 'owner' field or check that your email matches a Bullhorn CorporateUser."
        })
```

## Dependency

CR9 must be implemented first. CR10 imports `resolve_caller` and `IdentityResolutionError` from `identity.py`.

## Affected User Stories

- **US-2**: Create a contact record linked to a company — `owner` becomes optional.
- **US-1**: Create a company record — `owner` becomes optional (if currently required).
- **US-4**: Resolve consultant name to CorporateUser — existing behaviour preserved when caller provides `owner`.

## Acceptance Criteria

1. `create_contact` succeeds without an `owner` field when the authenticated user's email resolves to a CorporateUser. The created record's `owner` is set to the resolved CorporateUser ID.
2. `create_contact` succeeds with an explicit `owner` field. The caller-provided owner is used, not the authenticated user.
3. `create_company` has the same behaviour as criteria 1 and 2.
4. If identity resolution fails and no `owner` is provided, the tool returns a clear error with the `identity_resolution_failed` error code and a hint to provide an explicit owner.
5. The Bullhorn PUT payload contains `owner: {"id": int}` regardless of whether the owner was auto-populated or caller-provided — no CorporateUser data leaks into the payload (consistent with CR3).
6. All existing tests pass. Tests that previously provided `owner` explicitly continue to work unchanged.
7. The `update_record` tool is not affected.
8. The `bulk_import` tool is not affected.

## Testing

### Unit tests for `create_contact`

- `test_create_contact_owner_auto_populated` — mock `resolve_caller()` returning `{"id": 42, ...}`, call `create_contact` without `owner` in fields, assert `client.create()` receives `owner: {"id": 42}`.
- `test_create_contact_explicit_owner_wins` — provide `owner: {"id": 99}` in fields, assert `resolve_caller()` is never called, assert `client.create()` receives `owner: {"id": 99}`.
- `test_create_contact_explicit_owner_name_wins` — provide `owner: "Maryrose Lyons"` in fields, assert `resolve_owner()` is called (not `resolve_caller()`), existing flow preserved.
- `test_create_contact_identity_resolution_fails` — mock `resolve_caller()` raising `IdentityResolutionError`, call without `owner`, assert error response with `identity_resolution_failed`.
- `test_create_contact_no_owner_required_validation_removed` — call without `owner` (with successful identity resolution), assert no "owner is required" error.

### Unit tests for `create_company`

- `test_create_company_owner_auto_populated` — same pattern as contact.
- `test_create_company_explicit_owner_wins` — same pattern as contact.

### Payload audit tests

- `test_create_contact_auto_owner_payload_no_leak` — when owner is auto-populated, assert the PUT payload contains only `owner: {"id": int}` and no other CorporateUser fields (firstName, lastName, email from the resolved identity must not appear).

### End-to-end tests

- `test_e2e_create_contact_no_owner_auto_populated` — mock token, CorporateUser query, ClientContact search (no duplicates), ClientContact PUT, ClientContact GET; call `create_contact({"firstName": "Jane", "lastName": "Doe", "clientCorporation": {"id": 1}})` without owner; assert response has `changedEntityId` and created record's `owner.id` matches the resolved CorporateUser.
- `test_e2e_create_contact_explicit_owner_overrides` — same setup but with `owner: {"id": 99}` in fields; assert created record's `owner.id` is 99, not the resolved identity.
