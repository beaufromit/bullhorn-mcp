# CR21: Notes Read Tools (`get_notes_for_entity`, `search_notes`)

## Summary

Fresh LLM sessions currently take 4–6 failed tool calls to answer "what are
the notes on candidate 169020". `search_entities(entity="Note", query=...)` is
the intuitive path but it doesn't work for subject-entity ID filters —
Bullhorn's `/search/Note` is Lucene over `comments`, not a join. The correct
route ("all notes for record X") lives at `/query/NoteEntity?where=targetEntityID=<id>`,
which is not surfaced anywhere in the current MCP. Production note bodies also
contain click-to-call telemetry tags (`[cc:<uuid>,...]`) that pollute every
phone-integration note and are meaningless to a human reader.

This CR adds:

1. `get_notes_for_entity` — the "give me this record's notes" tool, using the
   two-step `/query/NoteEntity` → batch `/entity/Note/<ids>` pattern documented
   in the Bullhorn support forums.
2. `search_notes` — the "find notes mentioning X" tool, wrapping `/search/Note`
   with client-side entity filtering.
3. A hard guard in `query_entities` that fails fast when `entity="Note"` and
   directs the caller to the right tool.
4. Docstring updates on `search_entities`, `get_candidate`, and `get_job` to
   steer agents away from the wrong path.

---

## Motivation

Bullhorn's Note model has an unusual dual identity:

- **Writing** notes: `PUT /entity/Note` (covered since Sprint 6 via `add_note`)
- **Reading all notes for a record**: `GET /query/NoteEntity?where=targetEntityID=<id>` —
  `NoteEntity` is the join table; `/query/Note` does not exist
- **Full-text note search**: `GET /search/Note?query=...` — Lucene over `comments`

No existing tool exposes the read path. The Bullhorn entity reference confirms
`Note` supports `/search` only (not `/query`), making `query_entities(entity="Note", ...)`
silently return an API error. Without `get_notes_for_entity`, every agent session
that needs note history reinvents the wheel through trial and error.

The click-to-call telemetry stripping is a quality-of-life fix: Bullhorn phone
integrations inject `[cc:uuid,number,number,inbound|outbound]` tags into the
`comments` field of every call-logged note. The tags carry no human-readable
information; not stripping them causes every call note to start with junk.

## Design

### `get_notes_for_entity`

**Two-step pattern (two API calls regardless of result count):**

1. `GET /query/NoteEntity?where=targetEntityID=<id>&fields=id,note&count=<limit>`
   → extract Note IDs from `note.id` on each row.
2. `GET /entity/Note/<id1>,<id2>,...?fields=<resolved_fields>` — batch read
   via new `BullhornClient.get_many("Note", ids, fields)`.

Step 1 uses `exclude_deleted=False` (the `isDeleted` column on `NoteEntity` is
a join-row state, not the Note's deletion state — filtering it here would
misclassify active Notes linked by a row marked inactive).

Step 2 fetches full Note bodies including `isDeleted`. Notes with `isDeleted=True`
are then dropped in Python unless `include_deleted=True`.

Click-to-call tags are stripped from `comments` by `_strip_cc_telemetry`; any
matched tags are moved to a sibling `call_metadata` list on the note dict.

Python-side sort by `order_by` (supports `-field` for descending) before return.

**Signature:**
```
get_notes_for_entity(
    entity: str,              # one of _NOTE_TARGET_ENTITIES
    entity_id: int,
    limit: int = 50,
    start: int = 0,
    fields: str | None = None,   # default: id,action,comments,dateAdded,commentingPerson(id,firstName,lastName),isDeleted
    order_by: str = "-dateAdded",
    include_deleted: bool = False,
) -> str
```

Returns a flat JSON array of note dicts.

### `search_notes`

Wraps `client.search("Note", query=query, ...)`. Entity filter applied
client-side after fetch (Lucene by subject-entity ID is unreliable on
`/search/Note`). Same telemetry stripping applied.

**Signature:**
```
search_notes(
    query: str,                     # Lucene over comments, e.g. "visa sponsorship"
    entity_filter: dict | None = None,  # {"type": "Candidate", "id": 169020}
    limit: int = 20,
    start: int = 0,
    fields: str | None = None,
) -> str
```

Returns a flat JSON array. Entity filter matches by comparing the incoming
`entity_filter["id"]` against the note's `personReference.id` (for
Candidate/ClientContact), `jobOrder.id`, `clientCorporation.id`,
`placements[].id`, `leads[].id`, or `opportunities[].id` depending on
`entity_filter["type"]`.

### `query_entities` guard

At the top of the tool body (before any client call), if `entity == "Note"`,
return `format_response({"error": "entity_not_queryable", ...})` with a hint
pointing to `get_notes_for_entity` or `search_notes`. Note is the **only**
entity in the supported set that Bullhorn restricts to `/search` only. All
other entities remain unrestricted.

The docstring's existing entity list gains a one-liner:
> Note: `entity="Note"` is not supported — use `get_notes_for_entity` for
> record-scoped reads or `search_notes` for full-text search.

### `BullhornClient.get_many`

```python
def get_many(self, entity: str, ids: list[int], fields: str | None = None) -> list[dict]:
```

Builds `GET /entity/<entity>/<id1>,<id2>,...?fields=<fields>`. Returns
`response["data"]` (Bullhorn returns a list for comma-separated ID requests).
Callers must guard against empty `ids` — the method returns `[]` immediately
without an API call if `ids` is empty.

## User stories

### Story 1 — Fetch all notes for a Candidate

**As a** recruiter,
**I want** to call one tool with a Candidate ID and get back all their notes,
**so that** I don't have to trial-and-error through entity search patterns.

**Acceptance criteria:**

- `get_notes_for_entity("Candidate", 169020)` returns a JSON array of notes.
- Default sort is newest-first (`-dateAdded`).
- Soft-deleted notes are excluded by default; `include_deleted=True` includes them.
- `call_metadata` key is present on each note that had `[cc:...]` tags; `comments`
  is clean of those tags.
- Invalid entity returns `{"error": "invalid_entity", ...}` before hitting Bullhorn.
- Empty result (no notes found) returns `[]` — not an error.

---

### Story 2 — Fetch notes for any supported entity type

**As a** recruiter,
**I want** `get_notes_for_entity` to work for all seven entity types that `add_note`
accepts,
**so that** I can fetch notes on jobs, placements, contacts, and companies
without separate tool coverage.

**Acceptance criteria:**

- Tool accepts all seven members of `_NOTE_TARGET_ENTITIES`.
- Pagination works: `limit` and `start` are forwarded to the `/query/NoteEntity` call.
- `order_by` accepts both ascending (`dateAdded`) and descending (`-dateAdded`) notation.

---

### Story 3 — Full-text note search

**As a** recruiter,
**I want** to search all notes for a phrase (e.g. "visa sponsorship"),
**so that** I can surface notes across the database by content without knowing
which record they're attached to.

**Acceptance criteria:**

- `search_notes("visa sponsorship")` returns notes containing that phrase.
- Optional `entity_filter={"type": "Candidate", "id": 169020}` narrows results
  to notes attached to that specific record.
- Same telemetry stripping applied.
- Soft-deleted notes are excluded by default (via `exclude_deleted=True` on
  `client.search`).

---

### Story 4 — Hard guard on `query_entities(entity="Note")`

**As an** MCP agent,
**I want** `query_entities` to fail fast with a clear error for `entity="Note"`,
**so that** I don't waste a round-trip on a guaranteed API error and get a
useful redirect immediately.

**Acceptance criteria:**

- `query_entities("Note", "comments LIKE '%foo%'")` returns
  `{"error": "entity_not_queryable", ...}` without calling Bullhorn.
- The error message names `get_notes_for_entity` and `search_notes` explicitly.
- All other `entity` values continue to work as before.

---

### Story 5 — Tests

**As a** maintainer,
**I want** full test coverage for the new client method and both tools,
**so that** API contract and business logic can't silently regress.

**Acceptance criteria:**

- `TestGetMany` in `test_client.py` — single id, multi id, empty list (no API
  call), 401-retry, error path.
- `TestGetNotesForEntity` in `test_server.py` — happy path, telemetry stripping,
  soft-deleted filtering (default + opt-in), invalid entity, empty result,
  `-dateAdded` and `dateAdded` sort ordering.
- `TestSearchNotes` in `test_server.py` — happy path, `entity_filter` post-match,
  no-match filter, API error.
- `TestQueryEntitiesNoteGuard` in `test_server.py` — Note guard returns error;
  non-Note entity still passes through.
- All existing tests pass.
