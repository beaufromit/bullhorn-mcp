# CR11: Per-user Identity Cache — Fix First-Writer-Wins Bug in Multi-user Deployments

## Summary

The identity cache in `identity.py` stores the resolved Bullhorn CorporateUser in a single module-level variable (`_resolved_caller`). This works correctly when the server runs as a per-user process, but is broken when the server runs as a shared HTTP instance (the current deployment model since Sprint 15 / FR-11). The first user to resolve their identity poisons the cache for all subsequent users — every `create_contact` and `create_company` call thereafter is stamped with the wrong owner.

The fix is to key the cache by the authenticated user's Entra `sub` claim (a stable, unique, per-user identifier already present in the token) instead of using a single shared slot.

## Problem

`identity.py` line 25:

```python
_resolved_caller: dict | None = None
```

`resolve_caller()` checks `if _resolved_caller is not None: return _resolved_caller` before reading the token. In a multi-user HTTP deployment, once any user has resolved their identity, all subsequent callers skip token reading entirely and receive the cached identity of the first user.

**Concrete impact:** Paul McArdle resolves his identity first. Fergal Keys connects and calls `create_contact`. The cache hits on Paul's record. Fergal's new contact is created in Bullhorn with Paul as owner, with no error or warning surfaced.

## Root Cause

The cache was intentionally designed as a single slot with this comment (line 22):

> "Acceptable because the server runs as a single-user service; one authenticated user per process."

That assumption is no longer valid. Sprint 15 added HTTP transport (`MCP_TRANSPORT=http`), which allows multiple consultants to connect to a single shared server process. CR9/CR10 were implemented after Sprint 15 without updating this assumption.

## Required Changes

### 1. Replace single-slot cache with a per-user dict in `identity.py`

**Before:**
```python
_resolved_caller: dict | None = None

def resolve_caller(client):
    global _resolved_caller
    if _resolved_caller is not None:
        return _resolved_caller
    ...
    _resolved_caller = results[0]
    return _resolved_caller
```

**After:**
```python
_caller_cache: dict[str, dict] = {}  # keyed by token claims["sub"]

def resolve_caller(client):
    token = get_access_token()
    if token is None:
        raise IdentityResolutionError("No authentication token available")

    claims = getattr(token, "claims", {}) or {}
    sub = claims.get("sub")
    if not sub:
        raise IdentityResolutionError("No 'sub' claim found in token — cannot key identity cache")

    if sub in _caller_cache:
        return _caller_cache[sub]

    email = claims.get("email") or claims.get("preferred_username")
    if not email:
        raise IdentityResolutionError("No email claim found in token")

    results = client.query(
        entity="CorporateUser",
        where=f"email='{email}'",
        fields="id,firstName,lastName,email",
    )

    if len(results) == 0:
        raise IdentityResolutionError(f"No Bullhorn CorporateUser found for email '{email}'")
    if len(results) > 1:
        raise IdentityResolutionError(
            f"Multiple Bullhorn CorporateUsers found for email '{email}' — expected exactly one"
        )

    _caller_cache[sub] = results[0]
    return _caller_cache[sub]
```

Key changes:
- `sub` is extracted first, before `email` — it is the cache key, and its absence is itself an error
- Cache lookup uses `sub in _caller_cache` rather than a `None` check
- `email` extraction (and the Bullhorn query) only happens on a cache miss

### 2. Update `_reset_caller_cache()` for tests

```python
def _reset_caller_cache() -> None:
    """Clear the per-user identity cache. Used in tests for isolation."""
    global _caller_cache
    _caller_cache.clear()
```

### 3. Update the module comment

Remove the "single-user service" caveat. The cache is now safe for multi-user HTTP deployments.

### 4. No changes to `server.py`, `client.py`, or any MCP tool

`resolve_caller(client)` signature is unchanged. All callers are unaffected.

## Token Claims Available

The `sub` claim was confirmed present in the Entra token in CR9 diagnostic testing:

```json
{
  "sub": "yBJP1xnVwrQy9mSAQRB-Urrn_KZlBp2ZIbHn0kDKRlY",
  "email": "beau@thepanel.com",
  ...
}
```

`sub` is the stable Entra object identifier scoped to the app registration. It will not change if a user's email or display name changes, making it a more reliable cache key than `email`.

## Failure Behaviour

If the token has no `sub` claim, `resolve_caller()` raises `IdentityResolutionError` immediately, before attempting email extraction or the Bullhorn query. This surfaces a configuration problem loudly rather than silently falling back to a shared identity.

## Acceptance Criteria

1. Two concurrent users with different Entra tokens each resolve to their own Bullhorn CorporateUser. Neither receives the other's identity.
2. The second call from the same user does not re-query Bullhorn (cache hit on `sub`).
3. If the token has no `sub` claim, `resolve_caller()` raises `IdentityResolutionError` with a clear message.
4. `_reset_caller_cache()` clears all cached identities (not just one slot). Tests calling it remain isolated from each other.
5. The `resolve_caller()` signature is unchanged. No callers in `server.py` require modification.
6. All existing 244 tests pass unchanged.
