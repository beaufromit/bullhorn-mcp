# CR15: Shortlist Candidates to a Job (JobSubmission Writes)

## Summary

Add two MCP tools — `shortlist_candidate` and `shortlist_candidates` — that create `JobSubmission` records in Bullhorn linking candidates to jobs at a configurable status (default: `"Shortlisted"`). The single-candidate tool is the primitive; the batch tool is a thin loop over it. Both auto-stamp `sendingUser` ("Added By") from the authenticated MCP user via the existing identity-resolution flow, pre-check for existing submissions to keep the operation idempotent, and accept an optional `fields` dict for additional `JobSubmission` fields. A new `shortlist_config.py` module exposes one environment variable (`BULLHORN_SHORTLIST_STATUS`) for per-instance status configuration, mirroring the pattern established by `joborder_config.py` in CR14.

No changes are required in `client.py`: `BullhornClient.create()` and `BullhornClient.query()` already work for arbitrary entity types including `JobSubmission`.

## Motivation

A user has reported that the connector is highly effective for **finding** candidates inside Bullhorn that match a job spec — the existing `list_candidates`, `search_entities`, and `query_entities` tools satisfy the discovery workflow well. The natural next step in a recruiter's workflow is to **shortlist** those candidates against the job: recording in Bullhorn that specific candidates are being considered for a specific role, attributed to the recruiter who took the action.

In Bullhorn, this is modelled as a `JobSubmission` entity linking a `Candidate` to a `JobOrder` with a `status` field. Out of the box, Bullhorn ships statuses such as `"New Lead"`, `"Internal Submission"`, `"Submitted"`, `"Placed"`, `"Rejected"`, but instances customise these. For the team that surfaced this request, the relevant status is `"Shortlisted"`, but the implementation must allow other instances to configure their own.

A separate concern that this CR resolves is correct attribution. By default, when an MCP write hits Bullhorn, the `sendingUser` ("Added By") field is populated with the API service account, not the human user who initiated the action via the MCP client. CR10 introduced the pattern of auto-stamping `owner` from the authenticated end user via `identity.resolve_caller()` for `create_contact`, `create_company`, and `create_job`. CR15 extends the same pattern to `sendingUser` on `JobSubmission`, so that the "Added By" column in Bullhorn reflects the recruiter who shortlisted the candidate.

## API research

Source: <https://bullhorn.github.io/Creating-a-Submission/>

- **Endpoint**: `PUT /entity/JobSubmission`
- **Required fields** in the PUT body:
  - `candidate`: `{"id": <int>}`
  - `jobOrder`: `{"id": <int>}`
  - `status`: string (must match a configured picklist value for the instance)
  - `dateWebResponse`: Unix timestamp in milliseconds
- **Response**: `{"changedEntityType": "JobSubmission", "changedEntityId": <int>, "changeType": "INSERT"}`
- **`sendingUser`** is a to-one association to `CorporateUser`. Defaults to the API service account if omitted. To attribute the submission to the calling user, the tool must stamp this field explicitly.
- **Duplicate behavior**: Bullhorn does not prevent multiple `JobSubmission` records for the same `(candidate, jobOrder)` pair. Duplicate suppression is the responsibility of the caller.

## Required Changes

### 1. Add `shortlist_candidate` MCP tool

**File:** `src/bullhorn_mcp/server.py`

Add a new tool:

```python
@mcp.tool()
def shortlist_candidate(
    job_id: int,
    candidate_id: int,
    status: str | None = None,
    fields: dict | None = None,
) -> str:
```

Behavior, in order:

1. **Validate inputs.** `job_id` and `candidate_id` must be positive ints. On failure return a structured error (`{"error": "invalid_argument", "message": "..."}`).
2. **Resolve status.** If caller supplied `status`, use it. Otherwise call `get_shortlist_status()` from `shortlist_config.py` (default `"Shortlisted"`).
3. **Build base payload.**
   ```python
   payload = {
       "candidate": {"id": candidate_id},
       "jobOrder": {"id": job_id},
       "status": resolved_status,
   }
   ```
4. **Merge caller-supplied `fields`.** Resolve label/alias keys via `get_metadata().resolve_fields("JobSubmission", fields)` first, then merge. Caller values win over base payload defaults (status is the only base default; if caller passes `status` in `fields` it should be ignored in favour of the dedicated `status` parameter — but the dedicated parameter already covers this, so this clause is only a docstring note).
5. **Auto-stamp `dateWebResponse`.** If not in the merged payload, set it to `int(time.time() * 1000)`.
6. **Auto-stamp `sendingUser`.** If not in the merged payload, attempt `resolve_caller()` from `identity.py`. On success: set `sendingUser = {"id": resolved_user.id}`. On `IdentityResolutionError` (stdio mode with no Entra token, or HTTP mode with an unmappable principal): log a `WARNING` and proceed without `sendingUser` — Bullhorn will default to the API service account.
7. **Duplicate pre-check.** Call:
   ```python
   existing = client.query(
       "JobSubmission",
       where=f"candidate.id={candidate_id} AND jobOrder.id={job_id}",
       fields=["id", "status", "dateAdded", "sendingUser"],
   )
   ```
   If `existing["data"]` is non-empty, return:
   ```json
   {
     "duplicate": true,
     "message": "Candidate already has a JobSubmission on this job; no new submission created.",
     "existing": <first existing record>
   }
   ```
   Do not call `client.create`.
8. **Create.** `client.create("JobSubmission", payload)`. Return `{"changedEntityId": ..., "changeType": "INSERT", "data": ..., "duplicate": false}` as JSON via `format_response`.

The tool returns a JSON-encoded string identical in shape to `create_job`'s return value, with an additional `duplicate: bool` field at the top level.

### 2. Add `shortlist_candidates` MCP tool

**File:** `src/bullhorn_mcp/server.py`

```python
@mcp.tool()
def shortlist_candidates(
    job_id: int,
    candidate_ids: list[int],
    status: str | None = None,
    fields: dict | None = None,
) -> str:
```

Loops over `candidate_ids`, calling the same internal logic as `shortlist_candidate` for each. Collects per-candidate outcomes and returns a structured summary:

```json
{
  "job_id": 123,
  "results": [
    {"candidate_id": 1001, "status": "created",   "submission_id": 9001},
    {"candidate_id": 1002, "status": "duplicate", "submission_id": 8876},
    {"candidate_id": 1003, "status": "error",     "error": "candidate_not_found"}
  ],
  "summary": {"created": 1, "duplicates": 1, "errors": 1}
}
```

Implementation note: refactor the core logic of `shortlist_candidate` into a private `_shortlist_one(job_id, candidate_id, status, fields, client, metadata, resolved_sending_user)` helper. Both public tools call it. The single-candidate tool returns the result of one call; the batch tool calls it in a loop. Identity resolution (`resolve_caller`) is done once at the top of `shortlist_candidates` and the resolved user is threaded into each `_shortlist_one` call — this avoids N redundant identity-cache lookups.

If `candidate_ids` is empty, return:

```json
{"job_id": 123, "results": [], "summary": {"created": 0, "duplicates": 0, "errors": 0}}
```

Exceptions raised inside `_shortlist_one` (e.g. `BullhornAPIError`) are caught and recorded as `{"status": "error", "error": str(e)}` entries; the batch as a whole does not abort on a single failure.

### 3. Add `shortlist_config.py` module

**File:** `src/bullhorn_mcp/shortlist_config.py` (new)

Mirrors the shape of `joborder_config.py` but with a single loader, since we only have one configurable concern in this CR:

```python
"""Per-instance JobSubmission configuration loaded from environment variables."""

import os

DEFAULT_SHORTLIST_STATUS = "Shortlisted"


def get_shortlist_status() -> str:
    """Return the configured JobSubmission status for shortlisting.

    Reads BULLHORN_SHORTLIST_STATUS; falls back to DEFAULT_SHORTLIST_STATUS.
    """
    return os.environ.get("BULLHORN_SHORTLIST_STATUS", DEFAULT_SHORTLIST_STATUS)
```

The module is intentionally separate from `joborder_config.py` to keep entity-scoped configuration cohesive. If future CRs add `BULLHORN_SHORTLIST_ALIASES` / `BULLHORN_SHORTLIST_REQUIRED` / `BULLHORN_SHORTLIST_DEFAULTS` (parallelling the JobOrder config triplet), they have a clean home.

### 4. Reserve `FIELD_ALIASES["JobSubmission"]` slot

**File:** `src/bullhorn_mcp/metadata.py`

Add an empty `JobSubmission` entry to `FIELD_ALIASES`:

```python
FIELD_ALIASES: dict[str, dict[str, str]] = {
    "ClientContact": {...},
    "JobOrder": {..., **get_joborder_aliases()},
    "JobSubmission": {},  # reserved for future per-instance aliases
}
```

This reserves the slot so `resolve_fields("JobSubmission", ...)` always works, and so a future `get_shortlist_aliases()` env loader can be wired in without changing `resolve_fields` or adding an `if entity in FIELD_ALIASES` guard.

### 5. Startup status validation

**File:** `src/bullhorn_mcp/server.py`

Add a one-shot validation at the same point where `get_metadata()` is first invoked (or via a deliberate startup hook called from `main()`):

1. Call `metadata.get_fields("JobSubmission")`.
2. Locate the `status` field descriptor in the response. Read its `options` list (Bullhorn returns picklist options as a list of `{value, label}` dicts under `options`).
3. If `get_shortlist_status()` is not present in `[o["value"] for o in options]`, log:
   ```
   WARNING: Configured BULLHORN_SHORTLIST_STATUS=<value> is not in the
   JobSubmission status picklist for this instance. Bullhorn may reject
   shortlist submissions. Valid values: <list>.
   ```
4. **Do not raise.** The server starts and the tools remain callable; Bullhorn's own error response is authoritative on the actual PUT.

Wrap the validation in a try/except so that metadata fetch failures (e.g. transient auth issues) do not block server startup. Log and continue.

Run validation **once per process**, gated by a module-level `_status_validated` flag. The per-session metadata cache means the picklist is re-fetched on cold start, which is the only time it needs re-checking.

### Acceptance criteria

1. `shortlist_candidate(job_id, candidate_id)` creates a `JobSubmission` with status defaulting to `BULLHORN_SHORTLIST_STATUS` (default `"Shortlisted"`).
2. Caller-supplied `status=` parameter overrides the env default.
3. Optional `fields` dict is merged into the payload after `metadata.resolve_fields("JobSubmission", fields)`.
4. `sendingUser` is auto-stamped from the authenticated MCP user via `resolve_caller()` when not in `fields`.
5. If `fields` contains `sendingUser`, the caller's value wins.
6. `dateWebResponse` is auto-stamped to current Unix ms when not in `fields`.
7. Before creating, the tool queries existing `JobSubmission` records for `(candidate_id, job_id)`. If one exists, the tool returns that record with `duplicate: true` and does not create a second.
8. `shortlist_candidates(job_id, candidate_ids)` accepts a list, calls the internal `_shortlist_one` helper for each, returns a structured response with per-candidate `status: "created" | "duplicate" | "error"` entries plus a `summary` count.
9. `shortlist_candidates` resolves the calling user's identity exactly once per batch, not per candidate.
10. Server startup logs a `WARNING` (not error) if the configured status is not in the JobSubmission status picklist for the instance.
11. Tool docstrings clearly describe behavior, parameters, return shape, and duplicate-pre-check semantics for LLM consumers.
12. Stdio mode (no Entra token) still works — `sendingUser` falls back to the API user with a logged warning, the rest of the operation proceeds.
13. All existing tests pass (currently 306).
14. New test count target: ~30 tests added (config + single + batch + startup + E2E).

## Out of scope

- **Status transitions.** No tool for promoting a `JobSubmission` from `"Shortlisted"` to `"Submitted"` / `"Interview Scheduled"` / `"Placed"` / `"Rejected"` / etc. That's a future CR (e.g. `update_submission` or `transition_submission`).
- **Removing or withdrawing** a shortlist entry. Future CR.
- **`owners` field auto-stamping.** Only `sendingUser` is auto-stamped in this CR. `owners` can still be set explicitly via `fields={"owners": [{"id": ...}]}`.
- **Persistent metadata cache across restarts.** The per-session cache is sufficient for now; a `refresh_metadata` MCP tool is not added.
- **JobSubmission alias / required / defaults env config triplet** (`BULLHORN_SHORTLIST_ALIASES` etc., parallelling JobOrder). The slot is reserved in `FIELD_ALIASES["JobSubmission"]` but no loader is added; defer until there is a concrete instance requirement.
- **Bulk-import of `(job, candidate)` pairs across multiple jobs** via `bulk_import`. The new batch tool only handles many candidates against one job, which is the common shortlist workflow.
- **Convenience helpers for resolving job/candidate by name** (e.g. `shortlist_candidate_by_name(candidate_name, job_title, company_name)`). The LLM can chain `list_companies` → `list_jobs` → `shortlist_candidate` using existing tools. We ship CR15 as scoped and observe whether the chaining feels clunky in practice before adding magic helpers.

## Implementation notes

**Why two tools instead of one with a list.** The single-candidate tool is the primitive — clearest semantics, simplest tests, easiest for the LLM to reason about when shortlisting one candidate. The batch tool exists to avoid N round trips when the agent is shortlisting a result set, and to amortise identity resolution and the duplicate query overhead across many candidates. Implementing batch as a thin wrapper over `_shortlist_one` means there's no semantic divergence: every per-candidate outcome in the batch is exactly what the single tool would have returned.

**Why pre-check for duplicates instead of letting Bullhorn create them.** Bullhorn happily creates multiple `JobSubmission` records per `(candidate, job)` pair. Without a pre-check, an LLM that retries a failed call (network blip, ambiguous response) would silently create duplicate shortlists, polluting the Bullhorn UI's submission list and skewing recruiter activity reports. The cost of one extra `query` per shortlist call is acceptable in exchange for idempotency. Future optimisation: batch the duplicate check into a single query filtered by `jobOrder.id` and `candidate.id IN (...)` — kept out of the initial implementation to keep the batch tool a clean loop over the single-candidate primitive.

**Why env-var status config with startup validation, instead of discovery.** Bullhorn's metadata endpoint can tell us *what* statuses exist for a `JobSubmission`, but not *which* one means "shortlist" for this team. That's a deployment-time policy decision — the same status value (`"Shortlisted"`, `"Long-listed"`, `"Pre-screen"`, etc.) may be configured differently across instances. The env var captures that decision explicitly. Startup validation against the actual picklist catches misconfigurations early without blocking server startup; Bullhorn's own response remains authoritative on actual PUT failures.

**Why not write to `.env` at runtime.** Hosted/HTTP deployments often have read-only filesystems, and `.env` is a development convenience rather than a runtime data store. Production env vars come from systemd, k8s, or docker-compose, where the file may not exist. Mutating config from inside the app blurs the deploy-time vs runtime boundary in ways that surprise future operators. If persistent metadata-derived config becomes a need later, a separate cache file (not `.env`) with an explicit `refresh_metadata` tool is the right shape — out of scope for CR15.
