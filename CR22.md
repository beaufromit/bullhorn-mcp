# CR22 — Fix Notes Retrieval Bugs (Post-CR21 Patch)

## Context

A Claude chat session attempting to read notes for candidate Barry Delaney
hit three distinct failures in our notes tooling introduced/broken in CR21
(Sprint 29). Two are real bugs in our code; one is a Lucene limitation we
should defend against.

Error report: `/home/kearkan/Downloads/bullhorn-notes-error-report.md`

---

## Root-Cause Diagnosis

### Bug 1 (Critical) — `get_notes_for_entity` sends a non-existent field name

Commit `f77e36b` (Sprint 29 review fix C1) added a `targetEntityType` filter
to the NoteEntity join-table query. The C1 review was correct that the filter
was missing, but the field name does not exist on NoteEntity. Bullhorn returns:

> "Where clause 'targetEntityType' at position 27 is not a valid field name."

The correct Bullhorn NoteEntity column is **`targetEntityName`**. The tool is
broken for every supported entity.

### Bug 2 — `search_notes` default `fields` includes `clientCorporation`, invalid on `/search/Note`

`_NOTE_DEFAULT_FIELDS` is shared by both tools. `get_notes_for_entity` calls
`/entity/Note/<ids>` (supports `clientCorporation`); `search_notes` calls
`/search/Note` (does NOT). Bullhorn returns:

> "Invalid field 'clientCorporation' at position 128."

### Bug 3 (defensive) — `search_notes` silently forwards `*` to an endpoint that rejects it

A bare `*` query yields `Bad Query: {0}` from Bullhorn. We should reject this
client-side with a helpful message pointing to `get_notes_for_entity`.

## Changes

### Change 1 — Fix the NoteEntity filter field name

**File:** `src/bullhorn_mcp/server.py`

```diff
-where=f"targetEntityID={entity_id} AND targetEntityType='{entity}'",
+where=f"targetEntityID={entity_id} AND targetEntityName='{entity}'",
```

**Test:** Add assertions to `test_query_calls_note_entity_with_correct_where`:
- `assert "targetEntityName" in where_arg`
- `assert "targetEntityType" not in where_arg`

### Change 2 — Separate default field sets for the two tools

**File:** `src/bullhorn_mcp/server.py`

Add `_NOTE_SEARCH_DEFAULT_FIELDS` constant (identical to `_NOTE_DEFAULT_FIELDS`
minus `clientCorporation`). Wire it into `search_notes` as the default. Update
docstring to match.

**Test:** Add test confirming `clientCorporation` does not appear in the
fields string passed to the search client call when no `fields` arg is given.

### Change 3 — Guard against `*` / empty queries in `search_notes`

**File:** `src/bullhorn_mcp/server.py`

Early-return a structured `invalid_query` error if query is blank or `"*"`,
with a hint to use `get_notes_for_entity` for record-scoped reads.

**Test:** `search_notes("*")` returns `invalid_query` error with no HTTP call.
