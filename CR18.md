# CR18: Dynamic Tool Descriptions from Bullhorn Metadata at Startup

## Summary

A live MCP session against `claude.ai` produced a bug report
(`bullhorn-mcp-error-report.md`) showing the LLM repeatedly used `title` as a
Candidate field — both in the Lucene `query` parameter and in the `fields`
list. Bullhorn rejected the call with:

```
API request failed: 400 - {"errorMessage":"Invalid field 'title' at position 23.","errorCode":400}
```

The Candidate entity stores job title as `occupation`. `title` is not a valid
Candidate field at all (unlike ClientContact, where it exists as a
salutation). The LLM had no way to know this because every `@mcp.tool()`
description in `server.py` is a static docstring with at most 2–4 example
fields — there is no enumeration of valid fields per entity.

Fix: at server startup, fetch `/meta/{entity}` for every supported entity and
append a compact field summary to the relevant tool descriptions before
`mcp.run()` exposes them. The LLM sees the authoritative field list (with
picklist values for status-like fields) from the first `tools/list` call.
Custom fields added in Bullhorn surface automatically after a server restart —
no code change needed.

The bug report also flagged "No approval received" from `search_entities` on
Candidate (Error 1). A follow-up live test reproduced success on the same
inputs, confirming a transient session-token blip. No code fix needed.

## Motivation

Three alternatives were rejected:

- **Server-side field substitution** (e.g. rewriting `title:` → `occupation:`).
  Hides API errors and creates a hidden coupling that breaks if Bullhorn's
  schema shifts.
- **Pre-send field validation**. Removes the error from its actual cause; the
  agent loses signal when the upstream API changes.
- **Static description updates per field**. Doesn't scale — every Bullhorn
  custom field would need a code change, and operators with different tenant
  configurations would need different forks.

The metadata module (`BullhornMetadata` / `client.get_meta()`) already fetches
`/meta/{entity}` for internal label resolution. The schema is there; it just
never reaches the LLM. This CR closes that gap once, for every tool, with no
ongoing per-field maintenance.

## Root cause walk-through

**`src/bullhorn_mcp/server.py`** — every tool description is a docstring on
the decorated function. FastMCP reads the docstring at decoration time
(module import) and never re-renders it. Example for `list_candidates`
(server.py:192–235):

```python
@mcp.tool()
def list_candidates(
    query: str | None = None,
    fields: str | None = None,
    ...
) -> str:
    """List candidate records...

    Example queries: skillSet:Python, lastName:Smith, status:Active
    """
```

The example list mentions four fields. The LLM has no inventory of valid
candidate fields, no warning that `title` is invalid for Candidate, and no
list of valid `status` values. The same shape exists in `list_jobs`,
`list_contacts`, `list_companies`, `search_entities`, and `query_entities`.

`get_entity_fields` (server.py:1102–1140) exists, but it's a per-call lookup —
it only helps if the LLM proactively calls it before every query, which it
does not.

## Proposed fix

### Where to inject

In `main()` (server.py:1427), immediately before `mcp.run(...)`:

```python
try:
    asyncio.run(enrich_tool_descriptions(mcp, get_client()))
except Exception as e:
    logger.warning(f"Could not enrich tool descriptions: {e}")
```

Reasons this is the right injection point:

- All `@mcp.tool()` decorators have already fired by import-completion. Every
  `FunctionTool` exists with its static docstring as a fallback.
- `FunctionTool.description` is a mutable pydantic field (verified
  empirically). Direct mutation is observed by every client from the first
  `tools/list` call.
- `get_client()` is network-free. The first network call happens inside
  `get_meta()`, which only runs after `main()` starts. Tests that
  `import bullhorn_mcp.server` continue to work with no side effects.
- `mcp.get_tool()` is async in `fastmcp` 3.2.4, so the enrichment step uses a
  single `asyncio.run()` call inside the sync `main()`.

### New module: `src/bullhorn_mcp/descriptions.py`

Module-level constants and the enrichment function:

- `SUPPORTED_ENTITIES: list[str]` — all entities to fetch:
  `["Candidate", "ClientContact", "ClientCorporation", "JobOrder",
  "JobSubmission", "Note", "Placement", "UserMessage", "CorporateUser"]`
- `PICKLIST_FIELDS_TO_EXPAND: set[str]` — field names whose `options[]` get
  inlined as `Valid values: ...`:
  `{"status", "employmentType", "category", "type", "source"}`. Conservative —
  keeps descriptions compact.
- `TOOL_ENTITY_MAP: dict[str, list[str]]` — maps each tool name to the
  entities its description should advertise:
  - `list_candidates` → `["Candidate"]`
  - `list_contacts` → `["ClientContact"]`
  - `list_companies` → `["ClientCorporation"]`
  - `list_jobs` → `["JobOrder"]`
  - `search_entities` → `SUPPORTED_ENTITIES`
  - `query_entities` → `SUPPORTED_ENTITIES`
  - `shortlist_candidate` / `shortlist_candidates` → `["JobSubmission"]`
  - `create_company` → `["ClientCorporation"]`
  - `create_contact` → `["ClientContact"]`
  - `create_job` / `update_job` → `["JobOrder"]`
- `build_entity_section(entity, fields) -> str` — renders one entity's fields
  as a compact section. Per-field format:
  `` - `name` (TYPE) — Label [required] [Valid values: a, b, c]``
- `enrich_tool_descriptions(mcp, client) -> None` — async. Builds the
  per-entity summary once, then iterates `TOOL_ENTITY_MAP` and mutates
  `tool.description`. Per-entity and per-tool try/except so one failure
  doesn't sink the rest.

### Modified: `src/bullhorn_mcp/metadata.py`

`BullhornMetadata.get_fields()` currently drops `options[]` during projection
(metadata.py:60–69). Extend the projection to retain `options` when present.
Existing callers don't reference `options`, so this is non-breaking. The
cache stores the projection with `options` included.

### Modified: `src/bullhorn_mcp/server.py`

- Add `import asyncio` and `from .descriptions import enrich_tool_descriptions`
  at the top.
- Inside `main()` before `mcp.run(...)`, wrap `asyncio.run(...)` in try/except
  and log a warning on failure.
- Do **not** call enrichment at module import. Test imports remain
  side-effect-free.

### New tests: `tests/test_descriptions.py`

- `test_build_entity_section_basic` — renders the standard fields including
  name, label, type, required.
- `test_build_entity_section_includes_picklist_options` — `status` field's
  `options[]` are inlined as `Valid values: ...`.
- `test_build_entity_section_omits_non_status_picklists` — picklist field not
  in `PICKLIST_FIELDS_TO_EXPAND` has no inline options.
- `test_enrich_tool_descriptions_appends_to_existing` — mock `mcp.get_tool`
  and `client.get_meta`; assert descriptions are extended (not replaced) and
  the original docstring text remains.
- `test_enrich_tool_descriptions_graceful_on_meta_failure` — one entity's
  `get_meta` raises; other entities still succeed; no exception propagates.
- `test_enrich_tool_descriptions_graceful_on_auth_failure` —
  `AuthenticationError` from the metadata fetch is logged and swallowed.

### Modified: `tests/test_metadata.py`

- `test_get_fields_includes_options_when_present` — regression test for the
  metadata projection change.

## Key reused utilities

- `BullhornMetadata` (src/bullhorn_mcp/metadata.py:44) — already does
  `/meta/{entity}` fetching, caching, and the projection we extend.
- `BullhornClient.get_meta()` (src/bullhorn_mcp/client.py:269) — single source
  for the `/meta` HTTP call; includes auth/refresh.
- `get_client()` (src/bullhorn_mcp/server.py:123) — lazy global client, safe
  to call at startup.
- `BullhornAPIError` / `AuthenticationError` — existing exception classes.

## Verification

**Unit tests:**

```bash
.venv/bin/pytest tests/test_descriptions.py -v
.venv/bin/pytest tests/test_metadata.py -v
.venv/bin/pytest  # full suite — current 369 + new tests passing
```

**Manual end-to-end:**

1. Run the server in stdio mode: `.venv/bin/python -m bullhorn_mcp.server`.
2. From an MCP client, inspect `list_candidates.description` — confirm it
   includes a dynamic field section listing `occupation`, `status`, etc.
3. Confirm `status` picklist values (`Active`, `Passively Looking`, …) appear
   inline.
4. Confirm `title` is **absent** from the Candidate field list — this is the
   proof-point for the bug report.
5. Run `search_entities(entity="Candidate", query="occupation:CIO", ...)` —
   confirm success.

**Failure-path verification:**

1. Point `BULLHORN_LOGIN_URL` at an unreachable host.
2. Start the server — confirm it boots, logs a warning, and tools register
   with their static docstrings.
3. Confirm a normal tool call still returns a proper auth/network error (not
   silent breakage).

**Startup latency check:**

- Time a boot-only invocation against the real Bullhorn instance. Expect ~3–4.5
  seconds added for the 9-entity fetch. If significantly slower, revisit (see
  Out of scope).

## Out of scope (deferred)

- **Concurrent `/meta` fetches** via `httpx.AsyncClient`. Saves ~1–2 s but
  adds complexity; defer unless startup latency becomes painful.
- **Periodic in-process refresh** (every N hours). Restart-driven refresh is
  the agreed cadence.
- **MCP resources for entity schemas** (Option C from design discussion). Not
  needed once descriptions carry the field info.
- **Bullhorn-side permission audit for `/search/Candidate`**. Error 1 was
  transient; revisit only if it recurs.
- **`pyproject.toml` ↔ `fastmcp` package mismatch**: `pyproject.toml` declares
  `mcp>=1.0.0` but `server.py` imports `from fastmcp import FastMCP`
  (`fastmcp` 3.2.4). Surface separately — not part of this CR.

## Risks

- **Startup-time auth failure on misconfigured deployments**: previously
  surfaced only on first tool call. Mitigated by graceful fallback — server
  still boots.
- **`fastmcp` API change between versions**: `FunctionTool.description`
  mutation and async `mcp.get_tool()` were verified against `fastmcp` 3.2.4.
  Pin or document the dependency.
- **Description size bloat**: 9 entities × full field lists could enlarge
  `search_entities` / `query_entities` descriptions. Mitigated by compact
  formatting (one line per field; picklist options only for the conservative
  `PICKLIST_FIELDS_TO_EXPAND` set). Monitor token cost after rollout.
