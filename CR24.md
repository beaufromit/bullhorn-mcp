# CR24 тАФ Fix `search_emails` sort field and isDeleted clause

## Problem

`search_emails` was failing at runtime with:

> Bad sort: unknown field smtpSendDate

The Bullhorn `/search/UserMessage` endpoint returns `fieldsFromIndex: false` on this tenant, meaning the Lucene index is not fully configured. `smtpSendDate` (client-supplied timestamp) is not indexed for sorting. `smtpReceiveDate` (server-side receive timestamp) is indexed and sortable.

A second latent bug: `BullhornClient.search()` defaults `exclude_deleted=True`, appending `AND isDeleted:0` to every Lucene query. `UserMessage` has no `isDeleted` field (confirmed via `/meta/UserMessage`), so this clause is invalid on this entity.

## Changes

### `src/bullhorn_mcp/server.py` тАФ `search_emails`

1. `sort="-smtpSendDate"` тЖТ `sort="-smtpReceiveDate"`
2. Add `exclude_deleted=False` to `client.search()` call
3. Update docstring: "smtpSendDate" тЖТ "smtpReceiveDate"

### `tests/test_server.py` тАФ `TestSearchEmails`

1. `test_search_emails_basic`: assertion `sort == "-smtpSendDate"` тЖТ `"-smtpReceiveDate"`

## Out of scope

The `since`/`until` Lucene date filter (`smtpSendDate:[lo TO hi]`) silently returns 0 results when the field is not indexed тАФ a known separate problem, not addressed here.
