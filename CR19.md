# CR19: Candidate Creation and CV Parsing

## Summary

End-to-end workflow testing surfaced that the MCP exposes 21 tools but **no
way to create a Candidate**. The full recruiter flow (add company → contact →
job → candidate → shortlist) breaks at the candidate step — `create_candidate`
does not exist. When a CV-upload workflow was proposed, it was clear that there
are also no tools for CV parsing or file attachment.

This CR adds:

1. `create_candidate` — chat-driven candidate creation (LLM supplies fields
   researched in conversation), with duplicate pre-check and owner stamping.
2. `find_duplicate_candidates` — explicit standalone dup-check by name/email.
3. `parse_cv` / `parse_cv_text` — parse a binary CV or pasted text against
   the Bullhorn resume-parsing endpoint, returning unsaved JSON for review.
4. `create_candidate_from_cv` — parse → dup-check → create candidate →
   write child records → attach CV file, in one operation.
5. `attach_cv` — attach a CV to an **existing** Candidate, re-parse it, diff
   the fields against the existing record, and let the user pick which fields
   to update before committing.
6. `candidate_config.py` — per-tenant defaults / required fields / label
   aliases, mirroring `joborder_config.py`.

---

## Motivation

### Why `create_candidate` doesn't exist yet

The original PRD focused on *reading* CRM data (search, list, get) for AI
assistants, then progressively added write tooling for companies, contacts, and
jobs as those entities came up in real workflows. Candidates were never a write
target in the earlier sprints. The gap only became visible during the first
full-workflow test prompt.

### Why the Bullhorn resume parser, not a third-party parser

Bullhorn's `/resume/parseToCandidate` endpoint (backed by Textkernel) is the
same parser used by Bullhorn's own UI upload flow. Results go through the same
taxonomy and field mapping that the Bullhorn team supports — using a different
parser and then writing to Bullhorn would create field-mapping drift. The API
also returns `candidateWorkHistory`, `candidateEducation`, and `skillList` in
one call, saving us from re-parsing those sub-entities separately.

### Why two input modes (binary base64 vs text)

MCP tool arguments are JSON — there is no streaming binary primitive. Binary
files must be base64-encoded. The two parse endpoints map cleanly onto this:

| Input | MCP arg | Bullhorn endpoint |
|---|---|---|
| PDF / DOC / DOCX | `file_b64` (base64 string) | `POST /resume/parseToCandidate` (multipart) |
| Pasted text or HTML | `content` (plain string) | `POST /resume/parseToCandidateViaJson` (JSON body) |

A filesystem-path approach was rejected: in `MCP_TRANSPORT=http` (hosted)
mode the LLM client and the MCP server do not share a filesystem.

### Why the two-call confirmation flow for `attach_cv`

MCP tool calls are stateless. When re-parsing a CV against an existing
Candidate record the user must be shown a diff (current vs proposed) and choose
which fields to update. A single-call model would either blindly overwrite all
fields (destructive) or write nothing and require a second tool call anyway.
The explicit two-call design (preview then commit with `fields_to_update=[...]`)
makes the confirmation step part of the API contract rather than an informal
convention.

## Key Bullhorn API facts

**`PUT /entity/Candidate`** (create)
- Per-tenant required fields from `/meta` are loosely enforced in practice
  (`firstName + lastName` is often enough), but we validate tenant-configured
  required fields explicitly before write.
- `name` is **read-only** — auto-computed from `firstName + lastName`. Must
  not be sent.
- **No `title` field**. Job title = `occupation`. Same `_strip_contact_title`
  pattern applies.
- **No `clientCorporation` FK**. Current employer = `companyName` (free text).
  Candidate ↔ Company association only exists via Placement/JobOrder.
- Nested refs: `owner: {id}`, `category: {id}`, `categories: {data: [{id}]}`,
  `candidateSource: {id}`, `primarySkills: {data: [{id}]}`.
- `source` (free text) ≠ `candidateSource` (association).

**`POST /resume/parseToCandidate`** (parse binary CV)
- Returns unsaved JSON: `candidate`, `candidateEducation[]`,
  `candidateWorkHistory[]`, `skillList[]`.
- Input: multipart/form-data, single file. Query params: `format`, optional
  `populateDescription`.
- Format support: official docs say `html|text` only; in practice
  `pdf/doc/docx` work on most tenants. Validate against target tenant.
- Sync — no job-poll pattern. Use generous client timeout.
- Same `BhRestToken` / `rest_url` as entity calls.
- Known issue (sdk-rest #226): parser can emit values longer than `/meta`
  `maxLength`. Truncate before write.
- Known issue (sdk-rest #146): CVs without an email address can fail to parse.
- Skills in `skillList` may lack a Bullhorn `id` (Textkernel taxonomy mismatch).
  Unmatched skills should flow into the `skillSet` free-text field.

**`POST /resume/parseToCandidateViaJson`** (parse text/HTML)
- JSON body input. Uses existing `BullhornClient._request` (JSON transport).
  No multipart code needed.

**`PUT /file/Candidate/{id}/raw`** (attach file)
- Multipart upload. Query params: `externalID`, `fileType`.
- Separate from parsing — always a dedicated call after Candidate is created
  or confirmed existing.

**`BullhornClient._request()` today** — JSON-only (httpx, no multipart).
  A `_request_multipart` sibling (or `parse_resume_file`, `attach_file`
  methods) must be added.

## User stories

### Story 1 — Create candidate from chat research

**As a** recruiter,
**I want** to add a new candidate with details I've gathered in a
conversation (name, current job title, current company, email, mobile, skills,
source),
**so that** I can capture chat-driven research into Bullhorn without manual
data entry.

**Acceptance criteria:**

- New `create_candidate` MCP tool.
- Required args: `firstName`, `lastName`. Optional: `occupation`, `companyName`
  (free text), `email`, `mobile`/`phone`, `skillSet`, `source`, `status`, plus
  arbitrary additional fields via a `fields` dict.
- Owner auto-stamped from authenticated MCP user via `resolve_caller`; accepts
  explicit `owner` arg with same ambiguity-resolution branch as `create_contact`.
- `title` field stripped with a warning in the response (job title goes in
  `occupation`).
- `clientCorporation` rejected with a clear error pointing to `companyName`.
- Tenant-configurable required fields enforced before write
  (`BULLHORN_CANDIDATE_REQUIRED` env, mirrors `BULLHORN_JOBORDER_REQUIRED`).
- Tenant-configurable defaults applied before caller fields
  (`BULLHORN_CANDIDATE_DEFAULTS`, caller wins on conflict).
- Label resolution via metadata cache (e.g. `"Job Title"` → `occupation`).
- Returns standard `changedEntityId` / `changeType` / `data` on success.

---

### Story 2 — Duplicate-check before creating a candidate

**As a** recruiter,
**I want** the system to warn me if a candidate matching the one I'm adding
already exists,
**so that** I don't create duplicate records.

**Acceptance criteria:**

- `create_candidate` runs a duplicate pre-check before insert (`force: bool =
  False`).
- Matching strategy: **email exact match** (highest signal) OR firstName +
  lastName fuzzy match using existing `score_contact_match` (which already
  works on arbitrary `firstName`/`lastName` dicts).
- If best match scores ≥ 0.50, returns `{"duplicate_found": True, "match":
  {confidence, category, record}, "message": ...}` and does not insert. The
  match record includes firstName, lastName, email, phone, occupation,
  companyName, dateAdded for LLM-side comparison.
- `force=True` bypasses the check and creates regardless.
- Standalone `find_duplicate_candidates(first_name, last_name, email=None)` tool
  for explicit checks without committing to a write.

---

### Story 3 — Parse a CV and preview the fields before saving

**As a** recruiter,
**I want** to upload a CV and see the parsed fields (name, contact info, work
history, education, skills) without anything being saved yet,
**so that** I can review or edit before committing a record.

**Acceptance criteria:**

- New `parse_cv(file_b64, filename, format)` MCP tool — binary path, calls
  `POST /resume/parseToCandidate` via new multipart client method.
- New `parse_cv_text(content, content_type="text/plain")` MCP tool — text path,
  calls `POST /resume/parseToCandidateViaJson` via existing JSON transport.
- Both return parsed `candidate`, `candidateEducation`, `candidateWorkHistory`,
  `skillList` **without writing anything to Bullhorn**.
- Both also return duplicate-check results (Story 2 logic) keyed by parsed
  email + name, so the LLM/user can decide create vs update before any write
  call is made.

---

### Story 4 — Parse a CV and create the candidate in one step

**As a** recruiter,
**I want** to upload a CV and have a candidate created from it with the CV file
attached,
**so that** the whole "add candidate from CV" flow is one action.

**Acceptance criteria:**

- New `create_candidate_from_cv` MCP tool — accepts `file_b64` + `filename` +
  `format` (binary) or `content` + `content_type` (text); one must be provided.
- Internal flow: parse → dup-check → branch:
  - **No duplicate (or `force=True`):** create Candidate → write child records
    (education, work history, primarySkills) → attach CV file via
    `PUT /file/Candidate/{id}/raw` (binary input only; text-only skips attach).
  - **Duplicate found:** return dup-check JSON with parsed fields included so
    the LLM/user can compare and decide. Nothing written. Response hints at
    `attach_cv` (Story 7) for the "yes, same person" path and `force=True` for
    the "different person" path.
- Truncates parsed values against `/meta` `maxLength` before write.
- Unmatched parsed skills (no Bullhorn `id`) appended to `skillSet` free-text;
  matched skills written via `primarySkills` associations.
- Returns created Candidate ID, child record IDs, and FileAttachment metadata.

### Story 5 — Tenant configuration for Candidate

**As an** integration owner,
**I want** to configure per-tenant required fields, defaults, and label aliases
for Candidate,
**so that** the tool works on our specific Bullhorn instance without code
changes.

**Acceptance criteria:**

- New `candidate_config.py` module, direct copy of `joborder_config.py`
  pattern.
- Env vars: `BULLHORN_CANDIDATE_DEFAULTS`, `BULLHORN_CANDIDATE_REQUIRED`,
  `BULLHORN_CANDIDATE_ALIASES` (all JSON strings).
- Aliases wired into `FIELD_ALIASES["Candidate"]` in `metadata.py`. Hardcoded
  base aliases include `{"job title": "occupation"}` (mirrors ClientContact).
- README and `.env.example` updated with the three new vars.

### Story 7 — Attach a CV to an existing candidate with per-field confirmation

**As a** recruiter,
**I want** to attach a CV to a candidate that already exists in Bullhorn AND
have updated information from the CV reflected on their record, with the option
to review and approve each proposed field change before it's committed,
**so that** when a candidate sends in a refreshed resume their Bullhorn record
stays current — but I stay in control of which fields actually get overwritten.

**Acceptance criteria:**

New `attach_cv` MCP tool, implemented as a **two-call confirmation flow**
(stateless — re-parses on the second call; parse is sync and CVs are small).

**Call 1 — preview (no `fields_to_update` arg):**
- Accepts `candidate_id`, `file_b64`, `filename`, `format`.
- Parses the CV via `POST /resume/parseToCandidate`.
- Fetches the existing Candidate via `client.get` with enough fields to diff.
- Diffs parsed fields against the existing record. For each scalar field that
  the parser populated and that differs from the current value, returns a row
  `{field, current, proposed}`.
- Also returns proposed new `candidateWorkHistory` entries (skipping collisions
  matched on employer + title + dates), new `candidateEducation` entries
  (matched on school + degree + dates), and skills (Bullhorn-matched +
  unmatched text to append to `skillSet`).
- **Nothing is written. The CV file is not yet attached.**
- Response:
  ```json
  {
    "preview": true,
    "candidate_id": N,
    "proposed_field_changes": [{"field": "occupation", "current": "...", "proposed": "..."}],
    "proposed_work_history": [...],
    "proposed_education": [...],
    "proposed_skills": {"matched": [...], "unmatched_to_skillset": [...]},
    "message": "Review the proposed changes. To commit, call attach_cv again with fields_to_update=[...] listing only the field names you want applied. Pass include_work_history=true / include_education=true / include_skills=true to commit those sections. The CV file will be attached only on the commit call."
  }
  ```
- The LLM presents this diff to the user; the user picks which fields to accept.

**Call 2 — commit (with `fields_to_update: list[str]`):**
- Same args as Call 1 plus `fields_to_update` (list of field API names to
  apply) and optional `include_work_history`, `include_education`,
  `include_skills` booleans (all default `False`).
- Re-parses the CV, applies **only** the listed field updates (truncating
  against `/meta` `maxLength`).
- Conditionally appends work history, education, and skill updates.
- Attaches the CV file via `PUT /file/Candidate/{id}/raw`.
- `force_all: bool = False` shorthand — if `True`, applies every proposed
  change without an explicit list (recruiter convenience).
- Returns:
  ```json
  {
    "committed": true,
    "candidate_id": N,
    "fields_updated": [...],
    "work_history_added": [...],
    "education_added": [...],
    "skills_added": {"matched": [...], "appended_to_skillset": [...]},
    "file_attachment": {id, name, contentType, dateAdded}
  }
  ```

This is the tool the LLM is steered toward when `create_candidate_from_cv`
returns `duplicate_found` and the user/LLM has confirmed the existing record
is the same person.
