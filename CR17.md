# CR17: Fix `shortlist_candidates` False Errors from `fields=*` on JobSubmission

## Summary

`shortlist_candidates` (and `shortlist_candidate` on a fresh write) reports
`status: "error"` for every candidate, yet the JobSubmission records are
created successfully in Bullhorn. The agent then retries, sees the records it
just wrote, and labels them duplicates.

Root cause: after a successful PUT, `BullhornClient.create()` fetches the
written record via `self.get(entity, entity_id)` to enrich its return value.
`get()` falls back to `fields="*"` when the entity has no entry in
`DEFAULT_FIELDS`, and `JobSubmission` has no such entry. The Panel's Bullhorn
tenant rejects `fields=*` on `JobSubmission` with:

```
400 - {"errorMessage":"You are not authorized to request all fields.",
       "errorMessageKey":"errors.allFieldsNotAllowed"}
```

The PUT succeeded; only the enrichment GET failed. The caller never sees the
new `submission_id`.

Fix: add a `JobSubmission` entry to `DEFAULT_FIELDS` containing only the
fields needed to confirm the write and label the response. The post-create
`get()` then requests a permitted subset instead of `*`.

## Motivation

The bug report (`bullhorn-shortlist-candidates-bug-report.md`) shows six
JobSubmissions all written (IDs 94607–94612, sequential `dateAdded`
timestamps), yet the batch tool reported `created: 0, errors: 6`. Callers
cannot trust the tool's response, which encourages retries that look like
duplicates.

This is not batch-specific — the singular `shortlist_candidate` has the same
bug for genuinely-new candidates. It only *appeared* to work in the report
because the singular call ran *after* the batch had already created the
record, so its duplicate pre-check short-circuited before reaching the
failing `get()`.

## Root cause walk-through

**`src/bullhorn_mcp/client.py:64`** — `create()`:

```python
def create(self, entity: str, data: dict[str, Any]) -> dict[str, Any]:
    result = self._request("PUT", f"/entity/{entity}", json=data)
    entity_id = result["changedEntityId"]
    record = self.get(entity, entity_id)          # ← enrichment
    return {"changedEntityId": entity_id, "changeType": "INSERT", "data": record}
```

**`src/bullhorn_mcp/client.py:175`** — `get()`:

```python
def get(self, entity, entity_id, fields=None):
    if fields is None:
        fields = DEFAULT_FIELDS.get(entity, "*")    # ← falls back to *
    params = {"fields": fields}
    result = self._request("GET", f"/entity/{entity}/{entity_id}", params)
```

**`src/bullhorn_mcp/client.py:10`** — `DEFAULT_FIELDS` (no `JobSubmission`).

**`src/bullhorn_mcp/server.py:1213`** — `_shortlist_one()` ends with
`result = client.create("JobSubmission", payload)` and returns the result.
Both `shortlist_candidate` and `shortlist_candidates` rely on this.

## Required changes

### 1. Add `JobSubmission` to `DEFAULT_FIELDS`

**File:** `src/bullhorn_mcp/client.py`

```python
DEFAULT_FIELDS = {
    ...
    "JobSubmission": (
        "id,status,dateAdded,"
        "candidate(id,firstName,lastName),"
        "jobOrder(id,title),"
        "sendingUser(id,firstName,lastName)"
    ),
    ...
}
```

Field list rationale:

- `id`, `status`, `dateAdded` — confirm the write, expose the `submission_id`
  the agent needs.
- `candidate(id,firstName,lastName)` — disambiguates in batch responses
  (matches the user's "id + candidate name is enough" guidance).
- `jobOrder(id,title)` — symmetric and cheap; useful when an agent
  cross-references the job.
- `sendingUser(id,firstName,lastName)` — the auto-stamped attribution
  (`_shortlist_one` already sets it on the payload); keeping it on the
  response lets callers verify it landed.

### 2. No call-site changes required

`_shortlist_one`, both shortlist tools, and the duplicate pre-check
(`client.query("JobSubmission", …, fields="id,status,dateAdded,sendingUser")`)
already work — they explicitly pass safe field lists. Only the implicit
enrichment fetch inside `client.create()` is changed by the new default.

### 3. Scope: JobSubmission only

Other entities without `DEFAULT_FIELDS` entries (e.g. `Note`) appear to work
on this tenant. Per `AGENTS.md` ("touch only what you're asked to touch"),
this CR fixes JobSubmission only. If the same class of error surfaces on
another entity, address it in a separate CR.

## Acceptance criteria

1. `DEFAULT_FIELDS["JobSubmission"]` exists with the field string specified
   above (no `*`).
2. `client.create("JobSubmission", payload)` issues the post-PUT GET with
   `fields=DEFAULT_FIELDS["JobSubmission"]`, never `fields=*`.
3. Calling `shortlist_candidates` against the Panel tenant with brand-new
   candidate IDs returns `status: "created"` and a `submission_id` for each
   candidate, with `summary.errors == 0`.
4. Re-running the same batch returns `status: "duplicate"` for each candidate
   with the matching `submission_id` (proves the first call actually
   succeeded and was correctly reported).
5. All 366 existing tests pass. New test count: ≥ 368.

## Out of scope

- Removing `fields="*"` as the universal fallback in `get()`. That fallback
  is the documented behaviour for ad-hoc reads of arbitrary entities; only
  the implicit enrichment path inside `create()` is the immediate risk and is
  fixed structurally by populating `DEFAULT_FIELDS` for the affected entity.
- Adding `DEFAULT_FIELDS` entries for other entities (`Note`, …) — out of
  scope until/unless a similar failure is observed.
- Adding a `fields=` parameter to `client.create()` so callers can override
  the enrichment fetch. Not needed for the reported bug and would be premature
  abstraction. Revisit if a future caller needs entity-specific field control.
- Refactoring `_shortlist_one` — no behavioural change needed.
