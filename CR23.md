# CR23 — Fix notes retrieval: replace NoteEntity two-step with entity association endpoint

## Problem

`get_notes_for_entity` returns `[]` for records that have notes. Verified against candidate 169020 (Barry Delaney): 6 notes confirmed in Bullhorn, tool returns empty array.

`search_notes` with `entity_filter` is also unreliable: the filter is applied client-side after a `/search/Note` Lucene query, but the Note Lucene index is not configured on all tenants (`fieldsFromIndex: false`), so the search itself returns 0 results before the filter is even reached.

## Root cause

### `get_notes_for_entity`

The current implementation queries the NoteEntity join table:

```
WHERE targetEntityID=169020 AND targetEntityName='Candidate'
```

Bullhorn stores `'User'` (not `'Candidate'`) in `targetEntityName` for Candidate-linked join rows — Candidate is a subtype of the `User` entity in Bullhorn's internal model. This filter never matches.

A naive fix (mapping `Candidate → 'User'`) is incomplete. Postman tests on a JobOrder's NoteEntity rows showed a mix of `targetEntityName='JobOrder'` and `targetEntityName='User'` rows for the same job — a targetEntityName filter would silently drop relevant notes regardless of the mapping.

### `search_notes` with `entity_filter`

Bullhorn's `/search/Note` operates on a Lucene index that is per-tenant configurable. On some tenants (including the one tested), the index contains no data (`fieldsFromIndex: false`), so all Lucene queries return 0. The current `entity_filter` path depends on `/search/Note` first returning results, then filters client-side — it cannot work when the index is empty.

## Fix

### `get_notes_for_entity`

Replace the two-step NoteEntity query → batch `get_many` flow with a single call to Bullhorn's entity association endpoint:

```
GET /entity/{Entity}/{id}/notes?fields=...&count=...&start=...&orderBy=...
```

Verified working via Postman for Candidate (all 6 notes returned) and JobOrder. Supports proper `count`/`start`/`orderBy` pagination server-side. No `targetEntityName` mapping required.

### `search_notes` with `entity_filter`

When `entity_filter` is provided, use the same entity association endpoint to fetch all notes for that record (up to 500), then apply Python substring search on the stripped `comments` field. Reliable on all tenants regardless of Lucene index configuration.

When no `entity_filter`, keep the existing `/search/Note` path unchanged — works on tenants with the index configured.

### `get_many` removal

`get_many()` was added in CR21 specifically for the NoteEntity two-step. After this CR it has no callers — remove it and its tests.

## Changes

### `src/bullhorn_mcp/client.py`

**Remove** `get_many()` (lines 406–432).

**Add** `get_association()` in its place:

```python
def get_association(
    self,
    entity: str,
    entity_id: int,
    association: str,
    fields: str | None = None,
    count: int = 20,
    start: int = 0,
    order_by: str | None = None,
) -> list[dict[str, Any]]:
    params: dict[str, Any] = {
        "fields": fields or "id",
        "count": min(count, 500),
        "start": start,
    }
    if order_by:
        params["orderBy"] = order_by
    result = self._request("GET", f"/entity/{entity}/{entity_id}/{association}", params)
    return result.get("data", [])
```

### `src/bullhorn_mcp/server.py`

**`get_notes_for_entity`**: replace the NoteEntity two-step body with a single `get_association` call. Remove "Sorting is applied in Python" comment — ordering now done server-side.

**`search_notes`**: add entity_filter fast-path before the existing `/search/Note` call. When `entity_filter` is valid, call `get_association`, filter deleted notes, keyword-filter comments in Python (case-insensitive substring), apply `start`/`limit` pagination, strip CC tags, return. Fallthrough to `/search/Note` when no entity_filter.
