# CR12: Fix MCP Session Expiry — Add offline_access Scope and Document Token Lifetime

## Summary

Users of the hosted MCP server (Claude.ai, ChatGPT) are forced to re-authenticate via Windows Hello / Microsoft SSO after a period of inactivity. This is caused by two related issues: the Entra OAuth flow does not request a refresh token (`offline_access` scope is absent), and the default Entra access token lifetime (1 hour) is shorter than a typical working session. When the access token expires, the MCP client has nothing to use for a silent re-auth and prompts the user to sign in again.

## Problem

In `server.py`, `_build_auth()` constructs an `OIDCProxy` with:

```python
required_scopes=["openid", "profile", "email"],
extra_authorize_params={
    "scope": "openid profile email",
},
```

`offline_access` is absent from both scope declarations. Without it, Microsoft Entra does not issue a refresh token — only a short-lived access token with a default lifetime of 1 hour. When that token expires, FastMCP's `OIDCProxy` has no refresh token to use for silent renewal, the session is considered unauthenticated, and the MCP client (Claude.ai or ChatGPT) forces the user through the full sign-in flow again.

This affects all users of the hosted HTTP deployment. Users on stdio (local desktop) are unaffected.

## Root Cause

The `offline_access` scope is the standard OAuth 2.0 mechanism for requesting a refresh token from Entra. It must be included in the authorization request scope string. It is currently absent from the `extra_authorize_params` passed to `OIDCProxy`.

## Required Changes

### 1. Add `offline_access` to the authorization scope in `server.py`

**File:** `src/bullhorn_mcp/server.py`

In `_build_auth()`, update `extra_authorize_params` to include `offline_access`:

```python
extra_authorize_params={
    "scope": "openid profile email offline_access",
},
```

This causes Entra to issue a refresh token alongside the access token. FastMCP's `OIDCProxy` can then use the refresh token to silently renew access tokens without prompting the user.

No change is required to `required_scopes` — that list controls what the server validates on incoming tokens, not what it requests during the authorization flow.

### 2. Update `.env.example` with a comment about session persistence

**File:** `.env.example`

Add a comment under the Entra block noting that `offline_access` is handled automatically by the server and does not need to be set by the operator. This prevents future confusion about why the scope does not appear in the env config:

```
# Microsoft Entra OAuth (required for remote HTTP deployment)
# Note: offline_access scope is requested automatically by the server
# to enable session persistence. No additional configuration is required.
ENTRA_TENANT_ID=
ENTRA_CLIENT_ID=
ENTRA_CLIENT_SECRET=
MCP_BASE_URL=
```

### 3. Update `README.md` with token lifetime guidance

**File:** `README.md`

Add a `### Session Persistence` subsection under the `## Hosted Deployment` section (or `## Hosted Authentication Model`) explaining:

- The server requests `offline_access` automatically, which enables refresh tokens and avoids repeated sign-in prompts during a working session.
- The default Entra access token lifetime is 1 hour. If users are still being asked to re-authenticate more frequently than expected, a token lifetime policy can be configured in the Entra tenant.
- Document the two approaches available:
  - **Via Azure Portal:** App registrations → the app → Token configuration → add an optional claim or configure a Token Lifetime Policy via Microsoft Graph.
  - **Via PowerShell (if tenant admin access is available):** a brief example showing the command to extend access token lifetime to 8 hours and assign it to the app registration.

Do not hardcode any tenant-specific values in the README. This is guidance only.

### 4. No changes to authentication logic, tools, or tests

The `offline_access` scope change affects only what is sent in the Entra authorization redirect. It does not change:
- How tokens are validated on incoming requests (FastMCP handles this)
- The `resolve_caller()` identity resolution logic
- Any MCP tool behaviour
- The Bullhorn OAuth flow (separate auth layer, unaffected)

## Scope Notes

This CR does not configure a token lifetime policy in the Entra tenant — that is an infrastructure change outside the codebase. The README update in task 3 provides the operator with the information needed to configure it independently.

If FastMCP's `OIDCProxy` does not implement refresh token handling (i.e. it does not use the refresh token to silently renew access), the `offline_access` scope addition is still correct and harmless — it will begin working automatically if FastMCP adds refresh support in a future release. In that scenario, the README guidance about token lifetime policy becomes the primary mitigation.

## Affected User Stories

- **US-22**: Run the server in HTTP mode for remote clients — sessions now persist across idle periods without requiring re-authentication.

## Acceptance Criteria

1. The authorization request sent to Entra includes `offline_access` in the scope parameter.
2. After the change, a user who authenticates once can continue using the MCP tools across the full working day without being prompted to re-authenticate via Windows Hello.
3. `.env.example` includes a comment clarifying that `offline_access` is handled by the server automatically.
4. `README.md` includes a `Session Persistence` subsection under the hosted deployment documentation explaining the `offline_access` scope, the default token lifetime, and how to extend it via a token lifetime policy if needed.
5. All 248 existing tests pass unchanged. No new tests are required — the scope change is a one-line string update with no testable server logic.

## Testing Notes

The `offline_access` behaviour cannot be meaningfully unit tested because it depends on Entra issuing a refresh token, which requires a live Entra tenant. Verify the fix manually:

1. Deploy the updated server.
2. Connect from Claude.ai or ChatGPT and authenticate.
3. Leave the connection idle for longer than 1 hour.
4. Invoke a tool. If the session persists without a re-authentication prompt, the fix is working.

If re-authentication still occurs after more than 1 hour, the access token lifetime policy should be extended in the Entra tenant as documented in the README.
