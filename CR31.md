# CR31 — Extend `update_record` to support Candidate

## Problem

`update_record` documents only `"ClientContact"` and `"ClientCorporation"` as valid entity types. There is no tool to update a Candidate record — consultants cannot update occupation, availability, salary expectations, contact details, or any other candidate field through the MCP.

## Solution

Update the docstring only. The implementation already fully supports Candidate:

- `_strip_contact_title` already strips `title` and `name` from Candidate payloads (added in a prior sprint for `create_candidate`).
- Name recomputation on `firstName`/`lastName` changes already checks `entity in ("Candidate", "ClientContact")`.
- `client.update()` is generic — no entity restriction.
- `get_metadata().resolve_fields()` is generic.
- `descriptions.py` already maps `update_record` to `SUPPORTED_ENTITIES`, which includes `Candidate`.

There is no code to write — only the docstring gate keeping LLMs from using it.

## Changes

### `src/bullhorn_mcp/server.py`

Update the `update_record` docstring:

- Change `entity` description from `"ClientContact" or "ClientCorporation"` to also include `"Candidate"`.
- Add a Candidate example.
- Note the `title` stripping behaviour for Candidate (mirrors the existing ClientContact note).

```
entity: "ClientContact", "ClientCorporation", or "Candidate"
```

New example to add:
```
- update_record("Candidate", 11234, {"occupation": "Head of Engineering", "dateAvailable": 1735689600000})
```

## Tests

New class `TestUpdateRecordCandidate` in `tests/test_server.py`:

- `test_update_candidate_basic` — calls `update_record("Candidate", 11234, {"occupation": "Head of Engineering"})`, verifies `client.update` called with `"Candidate"`, `11234`, and resolved fields.
- `test_update_candidate_strips_title` — passes `{"title": "Mr", "occupation": "CTO"}`, verifies `title` stripped from payload and response contains `warnings`.
- `test_update_candidate_name_recomputed` — passes `{"firstName": "John", "lastName": "Smith"}`, verifies `client.get` fetches current name fields and `name` is recomputed in the update payload.
- `test_update_candidate_api_error` — `client.update` raises `BullhornAPIError`; response contains `"ERROR:"`.

Expected test delta: **+4 tests** (569 → 573, assuming CR29 and CR30 land first).

## Risk

Near zero. All logic paths were already exercised by ClientContact tests. The only change visible to callers is that the tool description now advertises Candidate support.
