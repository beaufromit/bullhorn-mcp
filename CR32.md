# CR32 ‚Äî Tearsheet (Hotlist) Management

## Problem

Tearsheets (called Hotlists in some Bullhorn UI versions) are named candidate lists consultants build for client briefs and passive talent pools. Currently:

- Reading tearsheets is technically possible via `query_entities(entity="Tearsheet", ...)` only if the LLM infers the internal entity name ‚Äî not reliable.
- Creating tearsheets and adding/removing candidates are not possible at all. Bullhorn uses association endpoints (`PUT`/`DELETE` to `/entity/Tearsheet/{id}/candidates/{ids}`) which have no equivalent in the client.

## Solution

Add two client methods for association writes, five MCP tools, and register `Tearsheet` with the descriptions module.

## Assumptions

1. Tearsheet has an `isDeleted` field ‚Äî `exclude_deleted=True` applies for list queries. No change to `_ENTITIES_WITHOUT_ISDELETED`.
2. `create_tearsheet` auto-stamps `owner` from identity resolution (same pattern as `create_company`/`create_contact`). If identity resolution fails and no explicit owner is provided, the tool returns an `identity_resolution_failed` error and does not create the record.
3. `add_to_tearsheet` and `remove_from_tearsheet` accept a list of candidate IDs. Bullhorn's association endpoint accepts comma-separated IDs in the URL path, so multiple IDs resolve to a single API call.
4. Bullhorn returns HTTP 200 with a JSON body for both PUT and DELETE association operations. `_request` handles this without modification.
5. `get_tearsheet` fetches tearsheet metadata and candidates in two sequential API calls, returning them as a combined object. Candidate results are paginated via `get_association_with_meta`.
6. `Tearsheet` is added to `SUPPORTED_ENTITIES` in `descriptions.py` so the startup field-reference enrichment covers `list_tearsheets`, `get_tearsheet`, `create_tearsheet`, `add_to_tearsheet`, and `remove_from_tearsheet`.
7. No `update_tearsheet` tool in this CR ‚Äî renaming or re-describing a tearsheet is low frequency. Can be added to `update_record` in a follow-up if needed.

Add two new methods after `get_association`:

```python
def add_association(
    self,
    entity: str,
    entity_id: int,
    association: str,
    ids: list[int],
) -> dict[str, Any]:
    """Add records to a TO_MANY association via PUT.

    e.g. PUT /entity/Tearsheet/{id}/candidates/1,2,3
    """
    id_str = ",".join(str(i) for i in ids)
    return self._request("PUT", f"/entity/{entity}/{entity_id}/{association}/{id_str}")

def remove_association(
    self,
    entity: str,
    entity_id: int,
    association: str,
    ids: list[int],
) -> dict[str, Any]:
    """Remove records from a TO_MANY association via DELETE.

    e.g. DELETE /entity/Tearsheet/{id}/candidates/1,2,3
    """
    id_str = ",".join(str(i) for i in ids)
    return self._request("DELETE", f"/entity/{entity}/{entity_id}/{association}/{id_str}")
```

**`list_tearsheets`**

```python
@mcp.tool()
def list_tearsheets(
    query: str | None = None,
    limit: int = 20,
    start: int = 0,
    fields: str | None = None,
) -> str:
    """List tearsheets (hotlists) from Bullhorn.

    Args:
        query: Lucene query to filter tearsheets (e.g. 'name:CFO*' or 'owner.id:123')
        limit: Maximum number of results (1-500, default 20)
        start: Pagination offset (default 0)
        fields: Comma-separated fields to return
                (default: id,name,description,owner,dateAdded)

    Returns:
        JSON object with ``data`` (array of tearsheets) and ``pagination``.

    Examples:
        - list_tearsheets() - All tearsheets
        - list_tearsheets(query="name:CFO*") - Tearsheets whose name starts with CFO
        - list_tearsheets(query="owner.id:99") - Tearsheets owned by CorporateUser 99
    """
    try:
        client = get_client()
        meta = client.search_with_meta(
            entity="Tearsheet",
            query=query or "",
            fields=fields,
            count=limit,
            start=start,
            sort="-dateAdded",
        )
        return format_response(_paginate_envelope(meta, start, limit))
    except (AuthenticationError, BullhornAPIError) as e:
        return f"ERROR: {e}"
```

**`get_tearsheet`**

```python
@mcp.tool()
def get_tearsheet(
    tearsheet_id: int,
    candidate_limit: int = 200,
    candidate_fields: str | None = None,
) -> str:
    """Get a tearsheet and its candidate members by ID.

    Args:
        tearsheet_id: The Tearsheet ID
        candidate_limit: Maximum number of candidate members to return (default 200)
        candidate_fields: Comma-separated fields to return for each candidate
                          (default: id,firstName,lastName,email,occupation,status)

    Returns:
        JSON object with tearsheet metadata and a ``candidates`` key containing
        ``data`` (array of candidates) and ``pagination``.
    """
    try:
        client = get_client()
        default_candidate_fields = "id,firstName,lastName,email,occupation,status"

        tearsheet = client.get("Tearsheet", tearsheet_id)
        members_meta = client.get_association_with_meta(
            entity="Tearsheet",
            entity_id=tearsheet_id,
            association="candidates",
            fields=candidate_fields or default_candidate_fields,
            count=candidate_limit,
            start=0,
        )
        tearsheet["candidates"] = _paginate_envelope(members_meta, 0, candidate_limit)
        return format_response(tearsheet)
    except (AuthenticationError, BullhornAPIError) as e:
        return f"ERROR: {e}"
```

**`create_tearsheet`**

```python
@mcp.tool()
def create_tearsheet(
    name: str,
    description: str | None = None,
    owner: int | None = None,
) -> str:
    """Create a new tearsheet (hotlist) in Bullhorn.

    Args:
        name: Name of the tearsheet
        description: Optional description
        owner: Bullhorn CorporateUser ID to assign as owner.
               If omitted, the authenticated user is resolved automatically.

    Returns:
        JSON object with changedEntityId, changeType, and the created tearsheet record.
    """
    try:
        client = get_client()
        payload: dict[str, Any] = {"name": name}
        if description:
            payload["description"] = description

        if owner is not None:
            payload["owner"] = {"id": owner}
        else:
            resolved_owner = resolve_owner(client)
            if resolved_owner is None:
                return format_response({
                    "error": "identity_resolution_failed",
                    "message": "Could not resolve authenticated user to a Bullhorn CorporateUser. Provide owner explicitly.",
                })
            payload["owner"] = resolved_owner

        result = client.create("Tearsheet", payload)
        return format_response(result)
    except (AuthenticationError, BullhornAPIError) as e:
        return f"ERROR: {e}"
```

**`add_to_tearsheet`**

```python
@mcp.tool()
def add_to_tearsheet(tearsheet_id: int, candidate_ids: list[int]) -> str:
    """Add one or more candidates to a tearsheet.

    Args:
        tearsheet_id: The Tearsheet ID
        candidate_ids: List of Candidate IDs to add

    Returns:
        JSON object confirming the operation and listing the added candidate IDs.

    Examples:
        - add_to_tearsheet(tearsheet_id=55, candidate_ids=[101, 102, 103])
    """
    try:
        client = get_client()
        client.add_association("Tearsheet", tearsheet_id, "candidates", candidate_ids)
        return format_response({
            "tearsheet_id": tearsheet_id,
            "added": candidate_ids,
            "count": len(candidate_ids),
        })
    except (AuthenticationError, BullhornAPIError) as e:
        return f"ERROR: {e}"
```

## Risk

**Medium** (higher than recent CRs due to net-new client methods).

- `add_association` and `remove_association` use HTTP DELETE ‚Äî `_request` currently only allows 200/201 as success codes. If Bullhorn returns 204 on DELETE, the call will raise `BullhornAPIError`. Verify the actual Bullhorn response code for association DELETE during integration testing and adjust the success-code check in `_request` if needed.
- Tearsheet `isDeleted` assumption: if Bullhorn does not expose `isDeleted` on Tearsheet (similar to ClientCorporation), `list_tearsheets` queries will fail with a Bullhorn API error. If this happens, add `"Tearsheet"` to `_ENTITIES_WITHOUT_ISDELETED`.
- Identity resolution in `create_tearsheet` follows the same pattern as `create_company`/`create_contact` ‚Äî no new risk there.
