# CR28 — Surface pagination metadata on list/search/query tools

## Context

The Bullhorn REST API caps a single page at 500 records; most MCP tools default
to 20. The client methods `search()`, `query()`, and `get_association()` all
discard Bullhorn's response envelope (`total`, `start`, `count`) and return a
bare list. The LLM therefore has no way to know whether it received all
matching records or just the first page.

This CR surfaces that metadata so the LLM can tell when results are incomplete
and choose to paginate. Auto-pagination inside the server is deliberately out
of scope — task-driven completeness belongs in the caller's searching skill.

## Design

### Client (`src/bullhorn_mcp/client.py`)

Three new `*_with_meta` methods return the full envelope dict:
- `search_with_meta` → `{ data, total, start, count }`
- `query_with_meta` → `{ data, total, start, count }`
- `get_association_with_meta` → `{ data, total, start, count }`

The existing `search`, `query`, `get_association` become one-liners that call
the `_with_meta` variant and return `["data"]` — zero behavioral change for
the ~20 internal callers in `bulk.py`, `identity.py`, and dedup checks.

### Server (`src/bullhorn_mcp/server.py`)

New `_paginate_envelope(meta, start, count)` helper builds the user-facing
response dict with explicit `has_more` and `next_start` hints so the LLM
doesn't have to compute them.

Nine user-facing tools switch to `*_with_meta` + `_paginate_envelope`:

| Tool | Client method |
|---|---|
| `list_jobs` | `search_with_meta` |
| `list_candidates` | `search_with_meta` |
| `list_contacts` | `search_with_meta` |
| `list_companies` | `search_with_meta` |
| `search_entities` | `search_with_meta` |
| `query_entities` | `query_with_meta` |
| `search_emails` | `search_with_meta` |
| `search_notes` | `search_with_meta` |
| `get_notes_for_entity` | `get_association_with_meta` |

### Response shape (all 9 tools)

**Before:** `[ {...}, ... ]`

**After:**
```json
{
  "data": [ {...}, ... ],
  "pagination": {
    "total": 1234,
    "start": 0,
    "count": 50,
    "has_more": true,
    "next_start": 50
  }
}
```

When Bullhorn omits `total` (rare), `has_more` falls back to
`len(data) == requested_count` and `total` is `null`.

## Files modified

- `src/bullhorn_mcp/client.py` — add `*_with_meta` methods; thin-wrap existing
- `src/bullhorn_mcp/server.py` — add `_paginate_envelope`; switch 9 tools
- `tests/test_client.py` — tests for new `_with_meta` methods + regression on bare methods
- `tests/test_server.py` — update envelope assertions for 9 tools

## Files NOT touched

- `bulk.py`, `identity.py` — internal; keep bare-list API
- `get_job`, `get_candidate`, `get_company` — single-record getters, no pagination
- `descriptions.py` — CR18 enrichment is orthogonal
