# CR34: Trim startup tool-description enrichment (context-weight reduction)

## Status: DRAFT (awaiting approval)

## Motivation

The startup enrichment in `descriptions.py` appends a `## Field reference` block to
each tool's description. Measured against the live tenant (verified on the deployed
server, commit `8362aef`, enrichment running cleanly with no errors), the result is:

- Total tool descriptions: **~111k tokens** (plus ~7k of parameter schemas = ~118k).
- The four generic tools each carry the full field lists of all 10 entities:
  `search_entities` ~12,970 tok, `query_entities` ~12,924, `update_record` ~12,859,
  `get_entity_fields` ~12,818. Those four alone are **~51.6k tokens**.
- Single-entity tools inline every field of their entity uncapped: Candidate (274
  fields) ~3.0k tok, Placement (295 fields) ~3.2k tok, repeated across every tool
  that touches that entity (Candidate's block appears on 7 tools).

This payload is loaded into model context whenever the Bullhorn connector is enabled,
including conversations that never touch Bullhorn. On a client that loads tools
eagerly (which we must assume for claude.ai and cannot control), it consumes ~118k of
the context window before the user types anything.

The enrichment is functioning as designed; the problem is that it is maximalist where
it should be selective. `get_entity_fields` already exists as the on-demand discovery
path, so the full field catalogue does not need to be preloaded.

## Goals

- Cut the tool-description payload from ~111k to roughly ~15k tokens without losing
  capability (full field discovery stays available via `get_entity_fields`).
- Keep the genuinely useful parts of enrichment: live field names, required-field
  flags, inlined picklist values, and the configured custom fields that matter.

## Non-goals

- No tool consolidation or removal (that is CR35).
- No change to runtime behaviour of any tool, only to its description text.
- No change to `BullhornMetadata` or the `/meta` fetch path.

## Design

Two ideas: (1) build each entity's inline field set from a curated selection rather
than dumping all `/meta` fields, and (2) give the generic tools a compact version of
that set instead of the all-entities dump.

### Field selection (`select_fields`)

New helper `select_fields(entity, meta_fields) -> list[dict]` returns a curated,
de-duplicated, ordered list drawn from the live `/meta` fields:

1. **DEFAULT_FIELDS** for the entity (`client.DEFAULT_FIELDS`), in declared order.
   These values are comma-separated strings and contain association syntax such as
   `candidate(id, name)`; parse the base name before `(` to match `/meta` (the field
   is `candidate`, type TO_ONE). Skip any name not present in `/meta`.
2. **Required fields** (`required: true`).
3. **Picklist fields** whose name is in `PICKLIST_FIELDS_TO_EXPAND`
   (`status, employmentType, category, type, source`).
4. **Configured custom fields**: fields whose name matches the Bullhorn custom-field
   pattern (`custom(Text|Int|Float|Date|Big)\d+`, plus `customObject*`) AND whose
   `label` differs from the API `name`. Rationale: an unconfigured custom field keeps
   its label equal to the API name, so `label != name` cleanly distinguishes the ones
   an admin has actually set up. Example confirmed live: `customText41` ->
   "Candidate Source - This Placement" (kept), `customText42` -> "Contributor" (kept),
   bare `customText40` -> "customText40" (dropped). (Note: a misconfigured field whose
   label is the prettified default like "Custom Text 43" will surface until its label
   is fixed in Bullhorn; accepted, since it pushes correctness to the source of truth.)

De-duplicate by name preserving first occurrence, then apply a safety cap
`MAX_FIELDS_PER_ENTITY = 40`. Order guarantees that if the cap trims anything, it trims
the lowest-priority custom fields, not required or picklist fields. Entities with no
`DEFAULT_FIELDS` entry (Note, CorporateUser) fall through to steps 2 to 4.

### Render levels (`build_entity_section`)

`build_entity_section(entity, meta_fields, level)` with two levels:

- **`level="full"`** (entity-specific tools): renders the full `select_fields` result
  with type, label, `[required]`, and inlined `[Valid values: ...]` for the picklist
  fields. Estimated ~300 to 500 tok per entity.
- **`level="compact"`** (generic tools): renders only the `DEFAULT_FIELDS` subset as
  `name (type) — label` lines, **no** picklist value expansion. Estimated ~1.2k per
  generic tool across all 10 entities.

### Tool wiring (`enrich_tool_descriptions` + `TOOL_ENTITY_MAP`)

- Entity-specific tools keep their `TOOL_ENTITY_MAP` entries and get `level="full"`.
- The four generic tools (`search_entities`, `query_entities`, `update_record`,
  `get_entity_fields`) get `level="compact"` for all `SUPPORTED_ENTITIES`, plus a
  trailing pointer line: `"For the full field list of any entity, call
  get_entity_fields(entity=\"<Entity>\")."`

### Base-docstring fallback (issue 3)

Add a one-line `get_entity_fields` mention to the **static** docstrings of the four
generic tools, so the guidance survives even if startup enrichment hits its
try/except fallback and appends nothing.

## Projected result

| Component                       | Before  | After (est.) |
|---------------------------------|---------|--------------|
| Generic tools (4, compact)      | ~51.6k  | ~5k          |
| Entity tool sections (full)     | ~59.4k  | ~10k         |
| **Description total**           | ~111k   | ~15k         |
| Parameter schemas               | ~7k     | ~7k (unchanged) |
| **Tool payload total**          | ~118k   | ~22k         |

Roughly an 80% reduction. Numbers are estimates until verified (see below).

## Behaviour change / risk

- Generic tools keep field **names** inline (compact level), so one-shot
  `update_record` / `query_entities` construction is preserved; only picklist values
  and the long tail move behind `get_entity_fields`.
- Entity tools surface configured custom fields automatically via the label heuristic,
  so newly-configured fields appear without a code change.
- The safety cap (40) can only trim low-priority custom fields; required and picklist
  fields are always retained.

## Tests (`tests/test_descriptions.py`, async)

- `select_fields`: includes DEFAULT_FIELDS (with composite names parsed to base),
  required, picklist, and `label != name` custom fields; excludes custom fields whose
  label equals the API name; dedupes; respects the cap with correct priority order;
  tolerates entities lacking DEFAULT_FIELDS.
- `build_entity_section`: `full` inlines picklist values and `[required]`; `compact`
  omits picklist values and lists only DEFAULT_FIELDS.
- `enrich_tool_descriptions`: generic tools contain the compact set plus the
  `get_entity_fields` pointer; an entity tool (e.g. `list_candidates`) contains the
  full set with a known required field and a `Valid values:` line.
- Audit existing description tests for field-presence assertions and repoint them.
- Graceful-fallback test unchanged; add an assertion that the generic-tool base
  docstrings mention `get_entity_fields`.
