# CR26 — Candidate & ClientContact `name` field auto-construction

## Problem

Bullhorn's REST API does **not** auto-compute the `name` field on Candidate **or** ClientContact
records when they are created/updated via PUT/POST. Unlike the UI (which concatenates firstName
+ lastName automatically), the API leaves `name` null if it is not explicitly included in the
payload.

This means records created through the MCP can have:
- `name = null` on the record
- The record **not visible in list views or appointments**
- No clickable name link in the Bullhorn UI — the record cannot be opened for editing

This has been confirmed as a known, reported issue in the Bullhorn support forums for both
entity types:
- https://supportforums.bullhorn.com/viewtopic.php?t=26648 (Candidate)
- https://supportforums.bullhorn.com/viewtopic.php?t=15384 (Candidate)
- https://supportforums.bullhorn.com/viewtopic.php?t=34545 (ClientContact — "Name on Contact not saving as expected")

`IMPLEMENTATION-PLAN.md:512` already contains an old note acknowledging this for ClientContact
("Bullhorn recommends sending `name` alongside `firstName`/`lastName` for ClientContact"), but
the code never acted on it — `create_contact` is a pure pass-through, so omitting `name` (the
common LLM case) silently breaks the record.

## Root cause in this codebase

- **Candidate:** `_strip_contact_title()` in `server.py` (lines 47–49) *removes* `name` from the
  Candidate payload if the caller sends it, on the false assumption Bullhorn auto-computes the
  field. Result: every Candidate created via the MCP has `name = null`.
- **ClientContact:** `create_contact` is a pure pass-through. If the LLM sends `"name": "Jane
  Doe"` it works; if it omits `name` (common case), the contact has `name = null` with the same
  downstream UI breakage. There is also no stripping, so an LLM-supplied `name` can drift from
  firstName/lastName.

## Design decision

`name` must **only** be set by the MCP, constructed from `firstName + " " + lastName` after field
resolution. It must never be passed through from LLM input — the MCP strips any caller-supplied
`name` value and replaces it with the computed one. This guarantees the stored value always
matches the component fields exactly.

This applies symmetrically to Candidate and ClientContact.

## Changes

### 1. `server.py` — new private helper

Add a single private helper used by all create/update paths:

```python
def _compute_person_name(fields: dict) -> str | None:
    """Return computed name from firstName + lastName, or None if neither is present."""
    first = str(fields.get("firstName") or "").strip()
    last = str(fields.get("lastName") or "").strip()
    combined = f"{first} {last}".strip()
    return combined or None
```

### 2. `server.py` — `_strip_contact_title()`

Move the `name`-stripping branch so it applies to **both** Candidate and ClientContact, with an
accurate warning message:

```python
if entity in ("Candidate", "ClientContact") and "name" in fields:
    fields = dict(fields)
    del fields["name"]
    msg = (
        f"Field 'name' was ignored on the {entity} payload — "
        "it is computed by the MCP from firstName + lastName to keep them in sync."
    )
    _logger.warning(msg)
    warnings.append(msg)
```

### 3. `server.py` — `create_candidate()`

After the `_strip_contact_title` call (currently line 1458) and before `client.create(...)`,
inject the computed name:

```python
computed = _compute_person_name(resolved)
if computed:
    resolved["name"] = computed
```

(`firstName`/`lastName` are validated as required earlier, so `computed` will not be None in
practice. The guard is defensive.)

### 4. `server.py` — `create_candidate_from_cv()` (two call sites)

Apply the same injection immediately before each `client.create("Candidate", ...)` call in
both CV-driven creation paths.

### 5. `server.py` — `create_contact()`

After `_strip_contact_title(resolved, "ClientContact")` and before
`client.create("ClientContact", resolved)`, apply the same injection.

Also update the `create_contact` docstring: remove the `"name": "Jane Doe"` strings from the
example payloads (currently around server.py:787 and :801) and add a one-liner: *"`name` is
computed by the MCP from firstName + lastName — do not send."*

### 6. `server.py` — `update_record()` for Candidate and ClientContact

In both the Candidate and ClientContact branches:
- After field resolution, check whether either `firstName` or `lastName` is in the resolved
  update payload.
- If **both** are present, compute name directly from the payload.
- If **only one** is present, fetch the current record (`client.get(entity, id,
  fields="firstName,lastName")`) to fill in the missing half, then compute.
- If **neither** is present, do nothing (no extra fetch).

Strip any LLM-supplied `name` from update payloads via the same `_strip_contact_title` path.

### 7. Tests

  - LLM-supplied `name="Bogus"` is stripped, computed value used, warning present
- `TestCreateCandidateFromCV`: same assertion for both CV paths
- `TestCreateContact` (new assertions):
  - `name` is in the create payload as `"First Last"`
  - LLM-supplied `name` is stripped + warning present
  - Remove any existing test that relied on `"name"` being passed through verbatim
- `TestUpdateRecord` (Candidate + ClientContact):
  - Update both firstName + lastName → `name` recomputed, no extra GET
  - Update only firstName → GET issued, `name` recomputed correctly
  - Update only lastName → GET issued, `name` recomputed correctly
  - Update neither → no recompute, no extra GET
  - LLM-supplied `name` in update payload is stripped
  - Other entities (e.g. ClientCorporation, JobOrder) — no name injection, no extra fetch

## What is NOT changing

- `create_company` — unaffected (`name` on ClientCorporation is the actual company name, not a
  derived field)
- CorporateUser `name` — unaffected (manually managed)
- No changes to `client.py`, `metadata.py`, `candidate_config.py`, `auth.py`, or `config.py`

## Risks

- Whitespace in firstName/lastName is normalised via `.strip()` — matches Bullhorn UI behaviour.
- The extra `GET` on partial-name updates adds one round trip when only firstName *or* lastName
  is updated. Low frequency in practice; acceptable cost for correctness.
- No change to existing `name` values on records created before this fix — only newly
  created/updated records are corrected. Historical records remain broken until separately
  backfilled (out of scope for this CR).
