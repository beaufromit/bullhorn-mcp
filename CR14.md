# CR14: Refactor create_job to Dict-Based Signature with Per-instance Configuration

## Summary

CR13's `create_job` cannot successfully create a JobOrder under any combination of inputs. The structured parameter signature defines names that do not correspond to real Bullhorn JobOrder API fields in the local instance (`website_sector_range`, `website_salary_range`, `website_location`, and a hardcoded `fee` that should be `feeArrangement`). When these parameters are null they fail required-field validation; when they are non-null they fail unknown-field validation. No caller input can satisfy both checks simultaneously.

CR14 fixes this by replacing the structured parameter design with a `fields: dict` signature matching `create_contact`, removing the broken validation infrastructure, and moving instance-specific concerns (field aliases, required fields, default values) to environment configuration. The hardcoded required-field list shrinks to Bullhorn's actual API minimum: `title`, `clientCorporation`, `clientContact`.

## Problem

The current `create_job` implementation has three structural issues stacked on top of each other.

**Structural issue 1: parameter names are not Bullhorn fields.** The function signature exposes `website_sector_range`, `website_salary_range`, `website_location` as Python parameters. These are not real JobOrder field names in the instance. The real fields are `customText1`, `customText10`, `customText11`. There is also a hardcoded `fee` parameter; the real Bullhorn field in this instance is `feeArrangement`.

**Structural issue 2: validation order makes the function uncallable.** `_missing_job_required_fields` rejects null values for the placeholder parameters before metadata resolution runs. `_validate_job_fields_known` rejects non-null values for the same parameters because they do not exist in metadata. There is no value the caller can pass that satisfies both gates. The full payload-by-payload investigation is documented in the chat log attached to the CR14 brief; nine attempts produced no successful create.

**Structural issue 3: structured parameters bypass discovery.** The `BullhornMetadata.resolve_label_to_api` lookup already maps Bullhorn UI labels to API names, but it only runs against keys passed in dict-shaped fields. When a parameter is exposed as a function argument, the caller must use that exact Python name, and metadata never gets a chance to remap it.

A separate concern raised during analysis is that instance-specific configuration (which fields are required, what their aliases are, what defaults apply when callers omit them) is currently hardcoded. This blocks portability and forces other Bullhorn instances adopting the MCP to inherit local business rules.

## Required Changes

### 1. Refactor `create_job` signature

**File:** `src/bullhorn_mcp/server.py`

Change the signature to:

```python
@mcp.tool()
def create_job(
    clientCorporation: dict,
    clientContact: dict,
    title: str,
    fields: dict | None = None,
) -> str:
```

`clientCorporation`, `clientContact`, and `title` remain explicit named parameters because they are Bullhorn's hard API minimum and the caller intent is clearer at the call site. Everything else flows through `fields`.

Remove:
- `source`, `grade`, `fee`, `salary`, `website_sector_range`, `website_salary_range`, `website_location`, `publicDescription`, `description`, `owner`, `status`, `isOpen`, `customText12`, `extra_fields` parameters
- `JOB_REQUIRED_BUSINESS_FIELDS` constant
- `_missing_job_required_fields` helper
- `_validate_job_fields_known` helper
- `_validate_job_reference` helper (replace with inline checks; see task 2)

`fields` is treated as a free-form dict the same way `create_contact` does. Aliases and defaults flow through it. Owner remains in `fields` and falls back to `resolve_caller()` when absent, identical to `create_contact`.

### 2. Inline relationship validation

Replace the removed `_validate_job_reference` with direct inline checks in `create_job`:

```python
if not isinstance(clientCorporation, dict) or "id" not in clientCorporation:
    return format_response({
        "error": "clientCorporation_required",
        "message": "clientCorporation must be an object containing an id.",
    })

if not isinstance(clientContact, dict) or "id" not in clientContact:
    return format_response({
        "error": "clientContact_required",
        "message": "clientContact must be an object containing an id.",
    })

if not isinstance(title, str) or not title.strip():
    return format_response({
        "error": "title_required",
        "message": "title must be a non-empty string.",
    })
```

These three are the only hardcoded required fields. Any further requirements come from environment configuration (task 3).

### 3. Add per-instance configuration loader

**File:** `src/bullhorn_mcp/joborder_config.py` (new module)

Create a new module that loads JobOrder configuration from environment variables. Three concerns, three env vars, all optional:

- `BULLHORN_JOBORDER_ALIASES`: JSON object mapping lowercase aliases to API field names. Extends the hardcoded `FIELD_ALIASES["JobOrder"]` at module load. Env entries take precedence on conflict.
- `BULLHORN_JOBORDER_REQUIRED`: JSON array of additional required fields beyond the hardcoded minimum. Each entry can be an API name or an alias; aliases are resolved before validation runs.
- `BULLHORN_JOBORDER_DEFAULTS`: JSON object of default field values applied when the caller does not supply them. Keys can be aliases or API names; resolution happens during merge.

Module shape:

```python
"""Per-instance JobOrder configuration loaded from environment variables."""

import json
import os
import logging

_logger = logging.getLogger(__name__)


def _load_json_env(var_name: str, default):
    """Load a JSON-encoded env var. Returns default on missing or invalid JSON."""
    raw = os.environ.get(var_name)
    if not raw:
        return default
    try:
        return json.loads(raw)
    except json.JSONDecodeError as e:
        _logger.warning(
            "Invalid JSON in %s, falling back to default. Error: %s", var_name, e
        )
        return default


def get_joborder_aliases() -> dict[str, str]:
    """Return env-defined JobOrder aliases (alias_lower -> api_field_name)."""
    raw = _load_json_env("BULLHORN_JOBORDER_ALIASES", {})
    if not isinstance(raw, dict):
        return {}
    return {k.lower(): v for k, v in raw.items()}


def get_joborder_required() -> list[str]:
    """Return env-defined additional required JobOrder fields."""
    raw = _load_json_env("BULLHORN_JOBORDER_REQUIRED", [])
    return list(raw) if isinstance(raw, list) else []


def get_joborder_defaults() -> dict:
    """Return env-defined JobOrder default field values."""
    raw = _load_json_env("BULLHORN_JOBORDER_DEFAULTS", {})
    return dict(raw) if isinstance(raw, dict) else {}
```

Validation behaviour: invalid JSON logs a warning and falls back to the empty default. The server does not refuse to start. This keeps the server resilient if an operator typos a config value.

### 4. Wire env aliases into `FIELD_ALIASES` at startup

**File:** `src/bullhorn_mcp/metadata.py`

`FIELD_ALIASES["JobOrder"]` should be extended at module load to include any env-defined aliases. The existing hardcoded entries (`"published description"`, `"public description"`, `"publish on website"`) remain. Env entries override hardcoded entries on key conflict so operators can correct mistakes without code changes.

```python
from .joborder_config import get_joborder_aliases

FIELD_ALIASES: dict[str, dict[str, str]] = {
    "ClientContact": {
        "job title": "occupation",
    },
    "JobOrder": {
        "published description": "publicDescription",
        "public description": "publicDescription",
        "publish on website": "customText12",
        **get_joborder_aliases(),
    },
}
```

The env load happens once at module import. Operators must restart the server to pick up alias changes. This is consistent with how all other env config is treated.

### 5. Apply defaults and required validation in `create_job`

**File:** `src/bullhorn_mcp/server.py`

The resolution order in `create_job`:

1. Build a working dict from caller `fields` plus the three named params (`clientCorporation`, `clientContact`, `title`).
2. Resolve owner via existing `resolve_caller()` fallback if not supplied, identical handling to `create_contact`.
3. Resolve the working dict through `metadata.resolve_fields("JobOrder", ...)` to API names.
4. Resolve env defaults through the same `resolve_fields` call.
5. Merge: `merged = resolved_defaults | resolved_caller_fields` so caller values always win on conflict.
6. Resolve env required list through the alias map to API names. Validate that all entries are present in `merged` and non-empty.
7. Call `client.create("JobOrder", merged)`.

Pseudocode:

```python
caller_fields = dict(fields) if fields else {}
caller_fields["clientCorporation"] = clientCorporation
caller_fields["clientContact"] = clientContact
caller_fields["title"] = title

# Owner fallback (identical to create_contact)
if "owner" not in caller_fields:
    try:
        caller = resolve_caller(client)
        caller_fields["owner"] = {"id": caller["id"]}
    except IdentityResolutionError as e:
        return format_response({
            "error": "identity_resolution_failed",
            "message": str(e),
            "hint": "Provide an explicit 'owner' field or check that your email matches a Bullhorn CorporateUser.",
        })
else:
    owner_result = client.resolve_owner(caller_fields["owner"])
    if isinstance(owner_result, list):
        return format_response({
            "error": "owner_ambiguous",
            "matches": owner_result,
            "message": "Multiple users found. Specify owner by ID.",
        })
    caller_fields["owner"] = owner_result

resolved_caller = metadata.resolve_fields("JobOrder", caller_fields)
resolved_defaults = metadata.resolve_fields("JobOrder", get_joborder_defaults())
merged = {**resolved_defaults, **resolved_caller}

# Validate env-defined required fields (hardcoded three already validated above).
# Run required entries through resolve_fields so operators can write aliases.
required_resolved = metadata.resolve_fields(
    "JobOrder", {k: None for k in get_joborder_required()}
)
missing = [
    k for k in required_resolved
    if k not in merged or merged[k] is None or merged[k] == ""
]
if missing:
    return format_response({
        "error": "required_fields_missing",
        "message": "Missing required JobOrder fields configured for this instance.",
        "fields": missing,
    })

result = client.create("JobOrder", merged)
return format_response(result)
```

Note: the trick with `{k: None for k in get_joborder_required()}` runs the required entries through the alias resolver so operators can write `"sector"` in `BULLHORN_JOBORDER_REQUIRED` and have it validated as `customText1`.

## Failure Behaviour

- Invalid JSON in any of the three env vars: log a warning and fall back to the empty default. Server starts normally.
- Env-defined alias points to a non-existent Bullhorn field: not validated at load time. Surfaces as a Bullhorn API error when create is attempted with a caller using that alias. Operators see this in error logs.
- Env-defined required field never gets supplied and has no default: every create call returns `required_fields_missing`. Visible immediately on first attempted create.
- Env-defined default value targets a non-existent field: surfaces as a Bullhorn API error on create. Same handling as alias misconfiguration.

## Affected User Stories

- Create a Bullhorn JobOrder from emails and user-confirmed details: currently broken in CR13, restored to working in CR14.
- Update a job's public description and related website fields: unaffected. `update_job` already uses dict-based input.

## Acceptance Criteria

1. `create_job` is callable with only `clientCorporation`, `clientContact`, and `title` and creates a JobOrder successfully (assuming env required list is empty and the instance accepts that minimal payload).
2. `clientCorporation` missing or malformed returns `clientCorporation_required` and does not call Bullhorn.
3. `clientContact` missing or malformed returns `clientContact_required` and does not call Bullhorn.
4. `title` missing or empty returns `title_required` and does not call Bullhorn.
5. `fields` accepts arbitrary keys, including aliases (`"sector"`, `"published description"`), API names (`customText1`, `publicDescription`), and metadata labels.
6. `BULLHORN_JOBORDER_ALIASES` extends `FIELD_ALIASES["JobOrder"]` at module load.
7. `BULLHORN_JOBORDER_DEFAULTS` values are applied to fields the caller does not supply, and caller values always win on conflict.
8. `BULLHORN_JOBORDER_REQUIRED` entries are validated as present in the merged payload after defaults are applied.
9. `BULLHORN_JOBORDER_REQUIRED` entries written as aliases resolve to API names before validation.
10. Owner falls back to `resolve_caller()` when not supplied, with `IdentityResolutionError` returning a structured `identity_resolution_failed` response. Caller-supplied owner always wins.
11. Invalid JSON in any of the three env vars causes a logged warning and falls back to the empty default. Server still starts.
12. `JOB_REQUIRED_BUSINESS_FIELDS`, `_missing_job_required_fields`, `_validate_job_fields_known`, and `_validate_job_reference` are removed from `server.py`.
13. README example for `create_job` reflects the dict-based signature.
14. `.env.example` includes the three new vars under a `JobOrder per-instance configuration` section.
15. All existing tests pass after CR13 tests are rewritten per task 9 below.

## Implementation Notes

- The `resolve_fields` call on the env defaults dict is a small extra cost on every create. Acceptable for the call frequency. If it shows up in profiling, defaults can be pre-resolved at module load.
- Operators introducing a new alias must restart the server. Consistent with how `.env` is treated elsewhere.
- The hardcoded `FIELD_ALIASES["JobOrder"]` retains the three `publicDescription`/`customText12` entries because they are useful for any Bullhorn instance with a similar custom-field setup. Env entries override them if needed.
- Owner handling stays in `create_job` and is not moved to a shared helper. The duplication with `create_contact` is small and the two flows have slightly different inputs (named params vs pure dict).
- `update_job` is not modified by CR14. Its dict-based signature already works correctly through the alias and metadata system.

## Out of Scope

- ClientContact and ClientCorporation env-driven aliases, defaults, and required-field configuration. The infrastructure (`joborder_config.py`) is JobOrder-specific in this CR. A future CR can generalise it if needed.
- JobOrder duplicate detection.
- Bulk job import.
- Auto-creation of `clientContact` or `clientCorporation` during job creation.
- Validation of env-defined alias targets at load time. Misconfigured aliases surface at create-time as Bullhorn API errors, which is acceptable given the operator-driven nature of the config.
