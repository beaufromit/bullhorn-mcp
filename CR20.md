# CR20: Pagination Support for List and Search Tools

## Summary

All list/search MCP tools are capped at a single page of results (max 500
records). There is no way to retrieve records beyond that window. The
underlying `BullhornClient.search()` and `BullhornClient.query()` methods
already accept a `start` offset parameter — it is simply not exposed through
any of the MCP tool signatures.

This CR adds an optional `start` parameter (integer, default `0`) to every
tool that calls a Bullhorn paginated endpoint and is currently missing it.

---

## Motivation

Bullhorn REST API pagination works via `start` (offset) and `count` (page
size). The client layer has supported both since Sprint 5 — the gap is
exclusively at the tool boundary. Recruiters with large databases (>500
candidates, >500 jobs) cannot retrieve records beyond the first page through
the MCP today.

---

## Scope

### Tools receiving `start`

| Tool | Underlying client call |
|---|---|
| `list_jobs` | `client.search("JobOrder", ...)` |
| `list_candidates` | `client.search("Candidate", ...)` |
| `list_contacts` | `client.search("ClientContact", ...)` |
| `list_companies` | `client.search("ClientCorporation", ...)` |
| `search_entities` | `client.search(entity, ...)` |
| `query_entities` | `client.query(entity, ...)` |

### Tools NOT in scope

All write tools (`create_*`, `update_*`, `add_note`, `shortlist_*`,
`bulk_import`, `attach_cv`, `parse_cv*`), the single-record getters
(`get_job`, `get_candidate`, `get_entity_fields`), `search_emails` (has
special `entityId` scoping — pagination is a separate concern), and
`find_duplicate_*` (internal fixed-count lookups; exposing `start` there
would be misleading).

---

## Design

### Parameter contract

- **`start: int = 0`** — zero-based record offset. Page 1 = `start=0`,
  page 2 = `start=500` (when `limit=500`), etc.
- **Existing `limit` parameter is unchanged** — controls page size, capped
  at 500 by the client.
- Backwards compatible — callers that do not pass `start` get the same
  behaviour as before (offset 0).

### Docstring update

Each affected tool's docstring receives:
- `start` in the Args section.
- A pagination example showing how `start` + `limit` combine (e.g.
  `start=0 limit=500` for page 1, `start=500 limit=500` for page 2).

### Client passthrough

`client.search()` and `client.query()` already forward `start` to the API
via the `params` dict. No client changes are needed.

## User stories

### Story 1 — Paginate candidate listings

**As a** recruiter,
**I want** to retrieve candidates beyond the first 500 results,
**so that** I can search large talent pools or export all records in batches.

**Acceptance criteria:**

- `list_candidates` accepts `start: int = 0`.
- Default behaviour (no `start` arg) is unchanged.
- Passing `start=500, limit=500` returns records 501–1000.
- The `start` value is forwarded to `client.search()` unchanged.

---

### Story 2 — Paginate all list/search tools consistently

**As an** MCP consumer,
**I want** every list/search tool to support `start`,
**so that** I can paginate any entity type without special-casing individual
tools.

**Acceptance criteria:**

- `list_jobs`, `list_contacts`, `list_companies`, `search_entities`, and
  `query_entities` all accept `start: int = 0`.
- All forward `start` to the underlying client call.
- Existing tests continue to pass (no default-value regression).

---

### Story 3 — Tests

**As a** maintainer,
**I want** tests confirming `start` is forwarded correctly,
**so that** pagination can't silently regress.

**Acceptance criteria:**

- Each affected tool gets at least one test that passes a non-zero `start`
  value and asserts it reaches `client.search()` / `client.query()` as
  the `start` kwarg.
- All existing tests pass.

## Critical files

| File | Change |
|---|---|
| `src/bullhorn_mcp/server.py` | Add `start: int = 0` param + docstring update to `list_jobs`, `list_candidates`, `list_contacts`, `list_companies`, `search_entities`, `query_entities`; pass `start=start` to the client call |
| `tests/test_server.py` | New test cases asserting `start` forwarding for each of the 6 tools |

No changes to `client.py`, `auth.py`, `config.py`, or any other module.

---

## Verification

1. `.venv/bin/pytest` — all existing tests pass; new pagination tests pass.
2. Manual smoke test against a Bullhorn sandbox:
   - `list_candidates(limit=10, start=0)` → first 10 records.
   - `list_candidates(limit=10, start=10)` → next 10 records, no overlap.
   - Confirm Bullhorn API request carries `start=10` in query params.
