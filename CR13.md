# CR13: Add First-class JobOrder Create and Update Tools

## Summary

Add first-class MCP tools for Bullhorn `JobOrder` writes:

- `create_job` — structured job creation for AI-assisted job setup from emails and user input.
- `update_job` — dedicated, documented JobOrder update path.

`update_record("JobOrder", ...)` currently works in practice because the implementation is generic, but it is not a good public contract. Its docstring says it is for `ClientCorporation` and `ClientContact`, the README says job writes are unsupported, and there is no job-specific validation, defaulting, or discoverable tool guidance.

CR13 makes job writes explicit, documented, and tested.

## Problem

The server can already read jobs through `list_jobs`, `get_job`, `search_entities(entity="JobOrder", ...)`, and `query_entities(entity="JobOrder", ...)`.

It cannot currently create jobs through a public MCP tool. There is an internal `BullhornClient.create(entity, data)` method, but no `create_job` wrapper and no JobOrder-specific validation/defaulting.

For updates, the generic `update_record` tool can currently be called with `entity="JobOrder"`, and tests confirm that JobOrder `title` is not stripped. However, this is accidental from the caller's perspective:

- The tool documentation says `update_record` updates `ClientCorporation` or `ClientContact`.
- README supported write targets do not include `JobOrder`.
- Callers have no job-specific guidance for fields such as `publicDescription` or `customText12`.

This blocks the intended workflow where an LLM reviews emails about a job, combines them with user input, creates a job in Bullhorn, and later writes back updates such as a published job description.

## Required Changes

### 1. Add `create_job` MCP tool

**File:** `src/bullhorn_mcp/server.py`

Add a new `@mcp.tool()` named `create_job`.

The tool should create a Bullhorn `JobOrder` by calling:

```python
client.create("JobOrder", resolved_payload)
```

The public API should be structured, not a raw fields-only API.

Minimum required business inputs:

- `clientCorporation`: Bullhorn company reference, `{"id": int}`.
- `clientContact`: Bullhorn client contact reference, `{"id": int}`.
- `title`: job title. The calling agent may generate this from emails/user input before calling the tool; the server should not invent one silently.
- `source`
- `grade`
- `fee`
- `salary`
- `website_sector_range`
- `website_salary_range`
- `website_location`

Optional inputs:

- `publicDescription`: published job description for Bullhorn's website/public job description field.
- `description`: internal job description/notes field, if provided by the caller.
- `owner`: explicit Bullhorn owner/consultant reference, `{"id": int}` or supported owner input if the existing owner resolver is reused.
- `status`
- `isOpen`
- `customText12`
- Additional JobOrder fields may be accepted via an `extra_fields` dict if the implementation chooses a hybrid signature, but required business inputs must remain explicit.

Defaults applied by the tool when omitted:

- `status`: `"Accepting Candidates"`
- `isOpen`: `True`
- `owner`: authenticated caller's Bullhorn CorporateUser via `resolve_caller()`
- `customText12`: `0` for "Publish on website"

Do not default `publicDescription`. Only send it when the caller provides published description text.

If identity resolution fails and no explicit `owner` is supplied, return the same style of structured error used by `create_contact`/`create_company`:

```json
{
  "error": "identity_resolution_failed",
  "message": "...",
  "hint": "Provide an explicit 'owner' field or check that your email matches a Bullhorn CorporateUser."
}
```

Caller-provided `owner` always wins.

### 2. Add `update_job` MCP tool

**File:** `src/bullhorn_mcp/server.py`

Add a new `@mcp.tool()` named `update_job`.

Signature should be:

```python
def update_job(job_id: int, fields: dict) -> str:
```

The tool should:

1. Resolve field labels/API aliases for `JobOrder`.
2. Send exactly and only the caller-specified fields after resolution.
3. Call `client.update("JobOrder", job_id, resolved_fields)`.
4. Return the existing update response shape:

```json
{
  "changedEntityId": 12345,
  "changeType": "UPDATE",
  "data": {}
}
```

`update_job` must not strip `title`. `title` is valid on `JobOrder`; the ClientContact salutation guard from CR7 must not apply to jobs.

`update_record` should remain unchanged for backward compatibility, but README/tool guidance should direct JobOrder callers to `update_job`.

### 3. Add JobOrder field aliases for known language gaps

**File:** `src/bullhorn_mcp/metadata.py`

Extend `FIELD_ALIASES` with a `JobOrder` entry.

Required aliases:

```python
"JobOrder": {
    "published description": "publicDescription",
    "public description": "publicDescription",
    "publish on website": "customText12",
}
```

The implementation should also support the tool's structured business field names through either metadata labels or explicit aliases:

- `source`
- `grade`
- `fee`
- `salary`
- `website_sector_range`
- `website_salary_range`
- `website_location`

Important: do not guess instance-specific Bullhorn API names for these fields if they are not already known. The implementation should use `get_entity_fields("JobOrder")` / metadata as the source of truth, then add aliases only for confirmed local Bullhorn field names.

If a structured field cannot be resolved to a known JobOrder field and has no confirmed alias, return a clear validation error before calling Bullhorn. Do not pass unknown structured keys through to Bullhorn silently for `create_job`.

### 4. Validate required relationships

`create_job` must validate that:

- `clientCorporation` is present and is a dict containing `id`.
- `clientContact` is present and is a dict containing `id`.

If either is missing or malformed, return a structured error and do not call Bullhorn.

The agent/user workflow is expected to identify or confirm the correct company and client contact before calling `create_job`. If there are multiple plausible contacts at the company, the agent should ask the user to confirm which Bullhorn contact record to use before creation.

### 5. Update README and tool documentation

**File:** `README.md`

Update the MCP tool list:

- Add `create_job` and `update_job` to write tools.
- Add `JobOrder` to supported write targets.

Add usage examples:

```text
create_job(
  clientCorporation={"id": 12345},
  clientContact={"id": 67890},
  title="Senior Software Engineer",
  source="Email",
  grade="Senior",
  fee=25000,
  salary=90000,
  website_sector_range="Technology",
  website_salary_range="80000-100000",
  website_location="London",
  publicDescription="Public-facing job description..."
)
```

```text
update_job(12345, {
  "publicDescription": "Updated public-facing job description...",
  "customText12": 0
})
```

Document that:

- `publicDescription` is Bullhorn's published job description field.
- `customText12` controls "Publish on website" in the local Bullhorn configuration.
- `customText12` defaults to `0`, meaning not published.
- Existing generic `update_record` remains available, but JobOrder updates should use `update_job`.

## Scope Notes

This CR does not add:

- Job duplicate detection.
- Bulk job import.
- Job deletion, merging, or archiving.
- Automatic creation of the company or client contact while creating a job.
- Automatic selection of a client contact when multiple plausible contacts exist.

The calling agent is responsible for using existing read tools (`list_companies`, `list_contacts`, `search_entities`, `query_entities`) to find or confirm company/contact records before creating the job.

## Affected User Stories

This extends the original product scope, which previously excluded JobOrder creation. It supports a new workflow:

- As a consultant or AI-assisted workflow, I want to create a Bullhorn JobOrder from emails and user-confirmed details, so that job opportunities can be set up without manually entering them in Bullhorn.
- As a consultant or AI-assisted workflow, I want to update a job's public description and related website fields, so that reviewed job content can be written back to Bullhorn.

## Acceptance Criteria

1. `create_job` is registered as an MCP tool.
2. `update_job` is registered as an MCP tool.
3. `create_job` calls `client.create("JobOrder", payload)` with a payload containing only validated/defaulted JobOrder fields.
4. `create_job` rejects missing or malformed `clientCorporation` before calling Bullhorn.
5. `create_job` rejects missing or malformed `clientContact` before calling Bullhorn.
6. `create_job` applies default `status: "Accepting Candidates"` when absent.
7. `create_job` applies default `isOpen: True` when absent.
8. `create_job` applies default `customText12: 0` when absent.
9. `create_job` auto-populates `owner: {"id": <caller_id>}` from `resolve_caller()` when owner is absent.
10. Caller-provided `owner` is preserved and not overwritten.
11. `publicDescription` is sent only when provided.
12. `"published description"` and `"public description"` resolve to `publicDescription` for JobOrder fields.
13. `"publish on website"` resolves to `customText12` for JobOrder fields.
14. Unknown structured create fields return a clear validation error instead of being silently passed through to Bullhorn.
15. `update_job` calls `client.update("JobOrder", job_id, resolved_fields)`.
16. `update_job` sends exactly and only caller-specified fields after resolution.
17. `update_job` can update `publicDescription`.
18. `update_job` does not strip or block `title`.
19. README documents the new tools and JobOrder write support.
20. All existing tests pass.

## Testing

### Unit tests for `create_job`

- `test_create_job_success` — mock metadata, identity resolution, and client create; call `create_job` with all required business inputs; assert `client.create("JobOrder", payload)` is called and response contains `changedEntityId`.
- `test_create_job_requires_client_corporation` — missing `clientCorporation` returns structured error and does not call `client.create()`.
- `test_create_job_requires_client_contact` — missing `clientContact` returns structured error and does not call `client.create()`.
- `test_create_job_rejects_malformed_client_corporation` — `clientCorporation` without `id` returns structured error.
- `test_create_job_rejects_malformed_client_contact` — `clientContact` without `id` returns structured error.
- `test_create_job_defaults` — omitted `status`, `isOpen`, and `customText12` result in `"Accepting Candidates"`, `True`, and `0` in the payload.
- `test_create_job_owner_auto_populated` — no owner supplied; mock `resolve_caller()` returning `{"id": 42}`; assert payload has `owner: {"id": 42}`.
- `test_create_job_explicit_owner_wins` — explicit owner supplied; assert `resolve_caller()` is not called and payload preserves caller owner.
- `test_create_job_identity_resolution_fails` — no owner supplied and `resolve_caller()` raises; assert `identity_resolution_failed` response and no create call.
- `test_create_job_public_description_optional` — omit `publicDescription`; assert it is not included in payload.
- `test_create_job_rejects_unknown_structured_field` — unresolved structured key returns validation error before API call.

### Unit tests for `update_job`

- `test_update_job_success` — calls `client.update("JobOrder", id, fields)` and returns formatted JSON.
- `test_update_job_label_resolution` — label/alias field resolves before update.
- `test_update_job_public_description_alias` — `"published description"` resolves to `publicDescription`.
- `test_update_job_publish_on_website_alias` — `"publish on website"` resolves to `customText12`.
- `test_update_job_does_not_strip_title` — `update_job(job_id, {"title": "Senior Engineer"})` sends `title` unchanged.
- `test_update_job_payload_only_contains_caller_fields` — update payload has exactly the resolved caller-provided keys.
- `test_update_job_api_error` — Bullhorn API error returns `"ERROR:"` response.

## Implementation Notes

- Reuse existing `BullhornClient.create()` and `BullhornClient.update()`; no new client methods are required unless the implementer finds a strong reason.
- Keep payload behavior consistent with CR2 and CR6: no unintended field injection.
- Keep ClientContact `title` stripping scoped only to ClientContact. JobOrder `title` is valid.
- Use Bullhorn metadata as the source of truth for JobOrder field validation wherever possible.
- If the local Bullhorn instance uses custom fields for source, grade, fee, sector range, salary range, or website location, document the confirmed API field names in `metadata.py` aliases and README examples during implementation.
