# CR16: Filter Soft-Deleted Records From All Searches

## Summary

Add a blanket `exclude_deleted=True` default to `BullhornClient.search()` and `BullhornClient.query()` so that soft-deleted Bullhorn records (`isDeleted:true`) are never returned from any search or query тАФ across all tools, now and in the future.

The client layer appends `AND isDeleted:0` (Lucene) or `AND isDeleted=false` (SQL WHERE) automatically. All existing call sites that already add `isDeleted:0` manually are cleaned up (the filter is now redundant at the call site). An `exclude_deleted=False` opt-out is available for the rare case where deleted records are genuinely needed.

No changes are required in `bulk.py` тАФ its three `self.client.search(...)` call sites automatically inherit the new default.

## Motivation

The duplicate-detection tools (`find_duplicate_companies`, `find_duplicate_contacts`) and the `create_contact` internal dedup pre-check sometimes report false positives: they flag a soft-deleted record as a potential duplicate, either blocking a new create or presenting deleted data to the agent.

The root cause is inconsistent filter application. The `list_*` tools defensively prepend `isDeleted:0` to their Lucene queries, but the dedup paths, the bulk-import paths, and the pass-through tools (`search_entities`, `query_entities`) do not. Bullhorn's `/search` and `/query` endpoints return deleted records by default тАФ the Lucene index includes them.

Fixing this call-site by call-site leaves the same trap for any future code path. Centralising the filter at `BullhornClient` means the fix is automatic for every present and future caller unless explicitly overridden.

## API research

Bullhorn's `/search/{entity}` endpoint uses Lucene syntax. The Lucene index includes all records regardless of their `isDeleted` value; callers must filter explicitly. The value `isDeleted:0` is the Lucene representation of the boolean false (Bullhorn indexes booleans as `0`/`1`). This is the existing convention used by the `list_*` tools.

Bullhorn's `/query/{entity}` endpoint uses a SQL-like WHERE syntax. The equivalent filter is `isDeleted=false`. This is already used in `_shortlist_one`'s duplicate pre-check query (`... AND isDeleted=false`).

Parenthesising the caller-supplied expression before appending the `AND isDeleted` clause is required to preserve boolean precedence. Without parentheses, a caller query of `name:Acme OR name:Globex` would parse as `name:Acme OR (name:Globex AND isDeleted:0)`, silently returning deleted Acme records.

## Required Changes

### 1. Add `exclude_deleted` parameter to `BullhornClient.search()` and `BullhornClient.query()`

**File:** `src/bullhorn_mcp/client.py`

Add `exclude_deleted: bool = True` to both methods. Implement query wrapping:

```python
# In search():
if exclude_deleted:
    query = f"({query}) AND isDeleted:0" if query else "isDeleted:0"

# In query():
if exclude_deleted:
    where = f"({where}) AND isDeleted=false" if where else "isDeleted=false"
```

Apply the wrapping **before** the `params` dict is built so the modified value flows through naturally.

### 2. Clean up `server.py`

**File:** `src/bullhorn_mcp/server.py`

**a) Remove redundant `isDeleted:0` fallbacks from list tools.**

`list_jobs`, `list_candidates`, `list_contacts`, `list_companies` each have:
```python
search_query = query or "isDeleted:0"
```
Replace with:
```python
search_query = query or "*:*"
```
Or simply pass `query` directly and let an empty/None query become `isDeleted:0` via the client layer. Since the tools already guard with `query or "isDeleted:0"`, the cleanest fix is to pass the user's query (or empty string) straight through тАФ the client will append `AND isDeleted:0` automatically.

The simplest refactor: change `search_query = query or "isDeleted:0"` to `search_query = query or ""` and pass `search_query` to `client.search()`. The client then handles the empty-query case (`isDeleted:0`).

**b) Delete `_company_name_search_query()`.**

This helper was introduced to build broad company search terms and inconsistently added `isDeleted:0` only for empty names. Now that the client handles all isDeleted filtering, the helper is only responsible for building the broad Lucene term тАФ and that logic (three lines) is duplicated inline in `find_duplicate_contacts` and `find_duplicate_companies` anyway. Delete the helper and inline the broad-term logic directly in both call sites:

```python
broad_term = name.split()[0] if name.strip() else name
# acronym handling: if all-caps word 2-6 chars, use first letter wildcard
if broad_term.isupper() and broad_term.isalpha() and 2 <= len(broad_term) <= 6:
    q = f"name:{broad_term[0]}*"
else:
    q = f"name:{broad_term}*"
```

**c) Simplify `_shortlist_one` WHERE clause.**

The `isDeleted=false` clause in the existing duplicate pre-check query:
```python
where=f"candidate.id={candidate_id} AND jobOrder.id={job_id} AND isDeleted=false"
```
becomes:
```python
where=f"candidate.id={candidate_id} AND jobOrder.id={job_id}"
```
The client appends `AND isDeleted=false` automatically.

**d) Update docstrings on `search_entities` and `query_entities`.**

Add a note in each docstring that soft-deleted records are excluded by default.

### 3. No changes to `bulk.py`

The three `self.client.search(...)` call sites automatically inherit the new default.

## Acceptance criteria

1. `BullhornClient.search(entity, query)` sends `(query) AND isDeleted:0` to Bullhorn when `exclude_deleted=True` (default).
2. `BullhornClient.search(entity, "")` sends `isDeleted:0` (no extra parens for empty query).
3. `BullhornClient.query(entity, where)` sends `(where) AND isDeleted=false` when `exclude_deleted=True`.
4. Both methods pass the query/where verbatim when `exclude_deleted=False`.
5. `find_duplicate_companies`, `find_duplicate_contacts`, `create_contact` dedup, `bulk_import` all exclude soft-deleted records without any call-site changes (they inherit the default).
6. `search_entities` and `query_entities` pass-through tools exclude soft-deleted records by default; agent-supplied `isDeleted:0` in the query is redundant but harmless.
7. `_company_name_search_query` is removed from server.py; its acronym/first-word logic is inlined at the two call sites.
8. `list_*` tools no longer hard-code `isDeleted:0` as a query fallback.
9. All 340 existing tests pass. New test count: тЙе 353.

## Out of scope

- A tool-level `include_deleted` parameter for `search_entities` / `query_entities`. If ever needed, the `exclude_deleted` flag is already in the client and can be surfaced then.

## Implementation notes

**Why parentheses around the caller query.** Without them, `name:Acme OR name:Globex AND isDeleted:0` would parse as `name:Acme OR (name:Globex AND isDeleted:0)` in both Lucene and SQL, silently returning deleted Acme records. Wrapping once unconditionally is simpler than analysing whether the caller query already has top-level disjunctions.

**Idempotency of the filter.** A caller that still passes `isDeleted:0` explicitly (e.g., an LLM using `search_entities`) produces `(isDeleted:0) AND isDeleted:0`. This is redundant but not an error.
