# CR33: Placement query fix and `list_placements` with new/extension awareness

## Status: COMPLETE

## Motivation

Two problems prevented useful Placement data access:

1. `query_with_meta` and `search_with_meta` unconditionally appended an `isDeleted`
   filter unless the entity was in a hard-coded denylist. Placement and
   PlacementChangeRequest have no `isDeleted` field, so every query returned
   HTTP 400 "Where clause 'isDeleted' is not a valid field name".

2. There was no tool to query placements, and no way to ask "what placements started
   in the last N months" while correctly excluding contract extensions. Investigation
   confirmed the extension data model: extensions are `PlacementChangeRequest` rows
   (requestType='Contract Extension') linked to one placement. The placement's
   `dateBegin` stays pinned to the original start; only `dateEnd` moves.

## Changes

### 1. Metadata-driven isDeleted gate (`src/bullhorn_mcp/client.py`)

- Added `self._isdeleted_cache: dict[str, bool] = {}` to `BullhornClient.__init__`.
- Added `_entity_has_isdeleted(entity)`: checks the static denylist first, then
  fetches `/meta/{entity}` and scans for the `isDeleted` field (cached per process).
  Falls back to `True` on any `/meta` error so existing behaviour is preserved.
- `search_with_meta` and `query_with_meta` now gate on `self._entity_has_isdeleted(entity)`
  instead of `entity not in _ENTITIES_WITHOUT_ISDELETED`.
- Updated `DEFAULT_FIELDS["Placement"]` to the richer default used by `list_placements`
  (dropped `salary`/`payRate`, added `employmentType`, `clientCorporation`, `customText41`).

### 2. `list_placements` tool (`src/bullhorn_mcp/server.py`)

New tool with `record_type` parameter selecting which start-event type to return:

| record_type  | Entity                 | Date field            |
|--------------|------------------------|-----------------------|
| "new"        | Placement              | dateBegin             |
| "extensions" | PlacementChangeRequest | requestCustomDate1    |
| "both"       | both (separately)      | per type above        |

Key behaviours:
- `since`/`until` ISO dates converted to epoch ms; `until` adds one day to include the full day.
- "extensions" always includes `requestType='Contract Extension'` in the WHERE.
- "new" defaults WHERE to `id IS NOT NULL` when no filters are supplied.
- Each row tagged with `record_type: "new"` or `"extension"`.
- "both" returns `{"new": {data,pagination}, "extensions": {data,pagination}}`; never merged.
- `status` param applies to "new" only (documented).
- Added `_iso_to_epoch_ms(date_str)` helper.

### 3. `descriptions.py`

Added `"list_placements": ["Placement"]` to `TOOL_ENTITY_MAP` so field metadata
is appended to the tool description at startup.
