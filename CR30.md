# CR30 â€” `get_job_submissions` tool

## Problem

There is no dedicated tool to fetch the submission pipeline for a job. A consultant asking "who is shortlisted to job 12345?" currently relies on the LLM inferring it should use `query_entities(entity="JobSubmission", where="jobOrder.id=12345")` â€” which works only if the model knows the Bullhorn entity schema. A dedicated tool makes this reliable across all models and removes dependence on LLM schema inference.

## Solution

Add `get_job_submissions(job_id, status=None, limit=20, start=0, fields=None)` to `server.py`. Internally uses `client.query_with_meta()` on `JobSubmission` filtered by `jobOrder.id`. Returns the standard CR28 paginated envelope (`data` + `pagination`). Add `"get_job_submissions": ["JobSubmission"]` to `TOOL_ENTITY_MAP`.

## Assumptions

1. Querying by `jobOrder.id=X` via `query_with_meta` (SQL WHERE syntax) is the correct approach â€” consistent with how `_shortlist_one` already queries JobSubmission.
2. Default fields: `id,candidate(id,firstName,lastName,email),status,dateAdded,sendingUser(id,name)`. These are the fields a consultant needs to see the pipeline at a glance.
3. `status` filter is optional. When provided, it appends `AND status="<value>"` to the WHERE clause. No validation against the picklist â€” callers get Bullhorn's error if they supply an invalid value.
4. `JobSubmission` is already in `SUPPORTED_ENTITIES` in `descriptions.py`. No change needed there.
5. `exclude_deleted` is handled automatically by `client.query()` for entities that support `isDeleted`. JobSubmission does have `isDeleted`; the default `exclude_deleted=True` should apply.
6. No new client method needed â€” `query_with_meta` already exists.

## Changes

### `src/bullhorn_mcp/server.py`

Add after `get_company` (alongside other `get_` tools, around line 562):

```python
@mcp.tool()
def get_job_submissions(
    job_id: int,
    status: str | None = None,
    limit: int = 20,
    start: int = 0,
    fields: str | None = None,
) -> str:
    """Get all candidate submissions (pipeline) for a specific job.

    Args:
        job_id: The JobOrder ID to fetch submissions for
        status: Filter by submission status (e.g., "Shortlisted", "Interviewing",
                "Offered", "Placed"). Omit to return all statuses.
        limit: Maximum number of results (1-500, default 20)
        start: Pagination offset (default 0). Use with limit to page through results.
        fields: Comma-separated fields to return
                (default: id,candidate(id,firstName,lastName,email),
                 status,dateAdded,sendingUser(id,name))

    Returns:
        JSON object with ``data`` (array of job submissions) and ``pagination``
        (``total``, ``start``, ``count``, ``has_more``, ``next_start``).
        When ``has_more`` is true, call again with ``start=<next_start>``
        to fetch the next page.

    Examples:
        - get_job_submissions(job_id=12345) - All submissions for job 12345
        - get_job_submissions(job_id=12345, status="Shortlisted") - Shortlisted only
        - get_job_submissions(job_id=12345, status="Interviewing", limit=50)
    """
    try:
        client = get_client()

        where = f"jobOrder.id={job_id}"
        if status:
            where += f' AND status="{status}"'

        default_fields = "id,candidate(id,firstName,lastName,email),status,dateAdded,sendingUser(id,name)"
        meta = client.query_with_meta(
            entity="JobSubmission",
            where=where,
            fields=fields or default_fields,
            count=limit,
            start=start,
        )

        return format_response(_paginate_envelope(meta, start, limit))

    except (AuthenticationError, BullhornAPIError) as e:
        return f"ERROR: {e}"
```

## Risk

Low. `query_with_meta` and `_paginate_envelope` are already exercised by 9 other tools. The only new logic is the WHERE clause construction, which is a simple string concatenation with no external calls.

One gotcha to watch: the `mock_client` fixture in `test_server.py` delegates `query_with_meta.side_effect` to `query.return_value` by default (CR28 pattern). Tests that directly set `query_with_meta.return_value` must also set `side_effect = None` on the mock â€” same pattern as existing CR28 tests.
